# Phase 5F UIリニューアル抜き出し計画

## 目的

このドキュメントは、補助エージェント Antigravity が作成したフェーズ5F UI改善差分をそのまま採用せず、良い要素だけをCodex側で安全に再実装するための計画である。

フェーズ5Fの目的は、既存機能を変えずに、社員へ初めて見せても安っぽく見えないモノトーン基調の業務アプリUIへ改善することとする。

## 基本方針

- Antigravity差分は「実装成果物」ではなく「UI改善案の素材集」として扱う。
- Codexの正規作業フォルダで、main起点の小さなPRに分けて再実装する。
- 5Fでは原則として機能追加をしない。見た目、余白、色、境界線、フォーカス表示、配置整理に限定する。
- 自動保存、重複判定、下書き、キャンセル、休み、祝日、コピーなどの既存仕様を変えない。
- 外部フォントや外部CDNへの依存は追加しない。
- 既存のテスト設定やDBまわりは、UI改善PRに混ぜない。
- READMEやロードマップの「完了」「RC確定」は、Codexレビューとテスト通過後にのみ更新する。
- UI改善PRでは、CSS変更が他画面へ波及する前提で全画面を確認する。
- Antigravity差分についての不採用理由は、Codexが差分とスクリーンショットを確認した事実に基づくものとして扱う。

## 守るDOM契約

入力フォームと一覧はJavaScriptやControllerと強く結びついているため、見た目改善でも以下を壊さない。

- `id`、`name`、`th:field`、`value`、`checked`、`selected`
- `action`、`method`、`href`
- `data-autosave-url`、`data-schedule-url`、`data-read-only` などの `data-*`
- hidden項目、`version`、`requestId`、`draftId`
- `required`、`disabled`、`hidden`、`readonly`
- `formnovalidate`
- 自動保存対象フォーム、一覧へ戻るボタン、下書き削除ボタン、依頼キャンセルボタンの識別に使う属性
- 既存JSが参照するクラス名やセレクタ

クラス名を整理したい場合は、新しい装飾用クラスを追加し、既存JSが依存しているクラスは残す。

## 共通確認ゲート

各5F PRは、少なくとも以下を確認してから採用する。

- `git diff --check`
- PC幅で対象画面のスクリーンショットまたは実ブラウザ目視確認
- 390px相当のiPhone幅で対象画面のスクリーンショットまたは実ブラウザ目視確認
- 文字はみ出し、重なり、固定月タブ干渉、横スクロール不能がないこと
- `ScheduleBrowserE2ETest`
- `ScheduleBrowserRecoveryE2ETest`
- `ScheduleVerticalSliceTest`
- 影響範囲が広い場合は全体テスト

## PR分割案

### 5F-1: スケジュール一覧の見た目改善

目的:

- 一覧性を落とさず、Excel感を弱める。
- ぱっと見で業務アプリらしい整った印象にする。

実装対象:

- カラートークンの整理
- ページヘッダーの圧縮
- 年月選択コントロールのピル型化
- スケジュール表の罫線、背景、余白の調整
- 案件セルのカード風表現
- 案件セルの左端カラーバー
- 継続セルの表示整理
- 過去日、休みセルのコントラスト改善
- 月タブの下線スリム化

実装しない:

- JSの挙動変更
- トースト通知
- scroll-snap
- 外部フォント
- README更新
- 入力フォームの大幅変更

既存システムとの整合性:

- 1セル30分単位の表構造は維持する。
- 1セル目に依頼者名、以降セルに矢印を表示する仕様は維持する。
- 5色の案件色割り当ては維持する。
- 休みセル、過去日セル、祝日除外の仕様は変更しない。
- 空白セルクリックで新規入力、入力済みセルクリックで詳細確認する仕様は維持する。

影響範囲:

- 主に `src/main/resources/static/css/app.css`
- 必要最小限で `src/main/resources/templates/schedule/month.html`

確認事項:

- PCで当月一覧が崩れないこと
- iPhone幅で横スクロールできること
- 入力済みセル、矢印セル、空白セル、休みセル、過去セルが判別できること
- 案件セル内の文字がはみ出しすぎないこと
- 下書き一覧の件数、リンク、理由表示が崩れないこと
- 月タブと固定表示が干渉しないこと

### 5F-2: 入力フォームの見た目改善

目的:

- 入力フォームを落ち着いた業務アプリ風に整える。
- 必須項目と任意項目を分かりやすくする。

実装対象:

実装しない:

- 文字数カウンター
- スピナー
- トースト通知

- フォーム構造の大幅変更

### 5Fバックログ: 挙動追加候補の再検討

目的:

- Antigravityが提案した挙動追加のうち、本当に必要なものだけを後続で検討する。

候補:

- トースト通知
- 送信ボタン内スピナー
- 文字数カウンター
- 未保存離脱警告

現時点の判断:

- 5F初回には含めない。
- 社員提示前に本当に必要かユーザーと再確認する。

## 受け入れ条件

5Fの各PRは、以下を満たした場合のみ採用する。

- 機能変更を含まない。
- `git diff --check` が成功する。
- 外部依存を追加しない。
- 自動保存、下書き、重複判定、キャンセル、休み、祝日、コピーの仕様を変えない。
- PC表示で主要画面が崩れない。
- iPhone幅で一覧と入力フォームが確認できる。
- 変更した共通CSSが、下書き一覧、キャンセル確認、コピー先選択、休み設定確認、404を崩していない。
- 必要なDOM契約を維持している。
- CSSキャッシュ回避用のクエリバージョンを更新する場合は、全テンプレートで整合している。
- 関連E2Eが成功する。
- 必要に応じて全体テストが成功する。

## 実装段階の結論

5Fは一括実装しない。

理由:

- 一覧、フォーム、周辺画面では壊れ方が違う。
- フォームは自動保存、下書き、STALE、キャンセルと結びついており、見た目変更でもリスクが高い。
- 共通CSSを一気に変えると、どの画面で崩れたか追いにくい。
- 社員提示前の品質を上げるには、画面ごとにレビューとスクリーンショット確認を挟む方が安全である。

推奨順:

1. 5F-1: スケジュール一覧
2. 5F-2: 入力フォーム
3. 5F-3: 周辺画面、スマホ、アクセシビリティ

挙動追加とテスト基盤改善は、5F初回実装には含めない。

## Codex側の採用判断

Antigravity差分は、以下の理由により直マージ不可とする。

- `git diff --check` 未実行かつ失敗状態だった。
- UI改善に挙動追加が混ざっていた。
- テスト設定変更がUI改善に混ざっていた。
- README、ロードマップで未レビューの完了、RC確定を記載していた。
- Google Fonts外部読み込みによりローカル業務アプリとしての安定性に懸念がある。
- `toast.js` に `innerHTML` 使用があり、将来的なXSSリスクがある。
- スティッキー操作領域が入力欄にかぶる可能性がスクリーンショット上で見えた。

ただし、UIの方向性自体は参考になるため、Codex主導で小さく安全に再実装する。
