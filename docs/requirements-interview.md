# 要件ヒアリング議事録

## このドキュメントの目的

顧客ヒアリングで出た要望、決定事項、上書きされた案、未確認事項を残す。会話の文脈が圧縮されても、ここを読めば最新の要件判断を復元できる状態にする。

このドキュメントでは、会話途中で考えが変わった箇所について、古い案を採用しないように明示する。

## 前提

- 顧客役: 現場で配送・設置案件を確認する担当者
- ベンダ役: 要件定義を支援するシニアPM
- 対象業務: コーヒーサーバー、ウォーターサーバーなどの配送・設置・回収・交換に関わるスケジュール管理
- 注意: 実在する会社名、人名、住所、電話番号、車両番号、顧客名は使わない。サンプルは必ず架空データにする。

## 最新の採用方針

### 1. システム化の目的

- 現行Excelの「ぱっと見で日付、時間帯、予定の埋まり具合が分かる」UIは気に入っている。
- そのため、Web版でもExcelに近いスケジュール表UIを重視する。
- 最大の課題は、Excelセル内に案件詳細を十分に書けないこと。
- Web版では、スケジュール表の見やすさを残しつつ、案件ごとの詳細入力フォームを用意する。
- 朝の確認、電話、LINE、口頭確認の手順を減らし、聞き漏れを防ぐ。
- 通知機能は便利だが、現時点では優先度は低い。

### 2. 現行の入力者と運用

- 入力する社員は日によって変わる。
- 関係する社員はおよそ10人前後。
- 1日に実際にスケジュールを埋める社員は3人程度。
- 複数人が入れたいタイミングで自由に入力する。
- 入力者と案件依頼者は同じ。
- 早い者勝ちで予定が埋まる。
- 最終承認者や確定担当者は置かない。
- 入力した時点で依頼として扱う。

### 3. スケジュール一覧UI

- 現行Excelに近い、日付と時間帯を軸にした表形式を採用する。
- 現行Excelは1セル1時間だが、新システムでは1セル30分とする。
- 表示時間帯は8:30-17:30固定とする。
- 8:30-17:30を30分単位で表示するため、1日あたり18セルを表示する。
- 案件の開始時間・終了時間も8:30-17:30内に限定する。
- 8:30より前、17:30より後の時間は入力不可とする。
- 1セルに入る案件は1件のみとする。
- 1案件は複数の30分セルにまたがることがある。
- サイトを開いた時点の現在年月の月間スケジュール一覧を自動表示する。
- 現行Excelの月別シートに近い骨組みを採用する。
- 月タブは、サイト表示時点の現在月を中心に、前月・当月・翌月の3か月分を表示する。
- 前月・当月・翌月以外の月を参照したい場合は、スケジュール一覧画面の左上端に置く年月選択の目次から、参照したい年度と月を選んで移動する。
- たとえば6月28日にサイトを開いた場合は6月版を表示し、7月予定を入れたい場合は月タブから7月版へ移動する。
- 月別スケジュール一覧の空白セルをクリックすると、その日付の新規入力フォームへ遷移する。
- 空白セルから開いた新規入力フォームでは、クリックしたセルの時間帯で入力時間を固定しない。
- たとえば一日中空いている日の14時セルをクリックし、入力フォームで10:00-12:00を指定しても問題なく反映される。
- 依頼者名、開始時間、終了時間がそろっている入力済みセルをクリックすると、そのセルに対応する既存案件の入力・編集フォームへ遷移する。
- 同じ案件が複数セルにまたがる場合、どの入力済みセルをクリックしても同じ案件の入力・編集フォームへ遷移する。
- その日の全セルが埋まっている場合、新規入力フォームへ移動できる空白セルは存在しない。
- フォームで入力された開始時間、終了時間に基づいて、該当する30分セルが同じ色で表示される。
- 先頭セルには依頼者名を表示する。
- 2セル目以降には矢印を表示する。
- 作業種別や依頼内容はセル内に表示せず、詳細フォームで確認する。
- 色はシステム側で自動割り当てする。
- 基本案は5色程度の固定パレットを用意し、案件の表示順に1色目から順番に割り当てる。
- ただし、よりシンプルな実装方法がある場合はそちらを採用してよい。
- 重要なのは、前後の案件で同じ色が連続せず、キャンセル後も一覧上で案件の時間範囲を見分けやすいことである。
- 利用者が色を選ぶ機能はMVPでは用意しない。

### 4. スケジュール表への反映条件

スケジュール一覧へ案件を表示する条件は、次の3つがそろっていること。

- 依頼者名が入力されている
- 開始時間が入力されている
- 終了時間が入力されている

この条件を満たす場合、該当日の開始時間から終了時間までの30分セルに色を付け、先頭セルに依頼者名、2セル目以降に矢印を表示する。

依頼者名、開始時間、終了時間のいずれかが未入力の場合、スケジュール一覧には反映しない。

依頼者名、開始時間、終了時間がそろって一覧に反映された後でも、依頼内容、現場住所、現場到着希望時間などの必須項目が未入力の場合は、先頭セルの依頼者名の下あたりに `＊未入力` と表示する。

入力・編集フォームを開いたとき、未入力の必須項目がある場合は、フォーム上部に赤文字で `依頼内容が未入力です。` のように項目名付きで表示する。

### 5. 時間範囲の重複チェック

- 同じ日の既存案件と時間範囲が重なる入力は無効にする。
- 重複時は保存せず、注意表示を出す。
- 表示例: `その時間はすでに埋まっています`
- 大規模な同時編集や入力衝突対策はMVPでは考慮しない。
- ただし、保存時または自動保存時の時間重複チェックは必要。
- `＊未入力` が表示されている案件でも、依頼者名、開始時間、終了時間がそろって一覧に反映されている場合は、通常の案件として時間枠を確保する。
- 未入力警告があっても、その時間帯は埋まっている扱いとし、別案件の重複入力はできない。
- 理由は、入力漏れがあっても依頼を取り損ねる方が客先トラブルにつながりやすいため。

例:

- 既存案件: 9:00-11:00
- 新規入力: 10:00-12:00
- 結果: 重複しているため反映しない

### 6. 入力フォームの保存方針

- 確定ボタンは置かない。
- 理由は、入力後に確定ボタンを押し忘れ、ページ移動で入力内容が消える事故を避けるため。
- 現行Excelに近く、入力した内容が反映される軽い操作感を重視する。
- 保存方式は「入力欄から離れたタイミングで自動保存」を採用する。
- ただし、明示的な確定ボタンを押さないと予定に反映されない方式は採用しない。

入力フォームに置く主なボタン:

- 依頼キャンセルボタン
- 前のページに戻るボタン

依頼キャンセルボタンは押し間違え防止のため、二重確認を表示する。

### 7. ステータス管理

MVPでは案件ステータスを持たない。

採用しないステータス:

- 確定
- 未確認
- 確認済み
- 作業中
- 完了
- キャンセル済み

理由:

- 1日の案件数は多くても数件程度で、完了管理の必要性が低い。
- 確認済みを置くと、その後に社員が変更・追記しづらくなる。
- キャンセル済みという表示は不要で、依頼キャンセルにより該当セルが空けば一覧上で判断できる。
- Excelライクな軽い運用を優先する。

キャンセル操作のMVP方針:

- 依頼キャンセルボタンで案件データを物理削除する。
- 削除後は該当セルを未入力扱いに戻す。
- 一覧上は該当セルの色と依頼者名が消える。
- キャンセル履歴の保持は将来拡張とする。

### 8. 作業種別

MVPの作業種別は次の4つとする。

- 設置
- 回収
- 交換
- 配達

作業種別によって入力フォームの項目は変更しない。設置、回収、交換、配達の細かい違いは、依頼内容欄にまとめて入力する。

### 9. 必須項目

MVPの入力フォームで必須にする項目:

- 依頼者名
- 開始時間
- 終了時間
- 作業種別
- 依頼内容
- 現場住所
- 現場到着希望時間
- 同行ありチェック

補足:

- 日付はフォームで選ばせない。スケジュール表でクリックした日付により決まる。
- 簡易案件タイトルは必須項目にしない。
- 台数、機種、内容物の詳細は独立項目にしない。
- 依頼内容という大きめの入力欄にまとめて記入する。

### 10. 条件付き必須項目

同行ありチェックを付けた場合のみ表示し、必須にする項目:

- 集合場所
- 出発時間
- 使用車両

理由:

- 同行なしの場合は、基本的に担当者本人の車を使う想定。
- 同行ありの場合は、集合場所、出発時間、使用車両がないと当日確認が必要になる。
- 同行ありを選んだ状態で集合場所、出発時間、使用車両のいずれかが未入力の場合も、通常の必須項目と同じく一覧の先頭セルに `＊未入力` を表示し、フォーム上部に赤文字で不足項目を表示する。
- 同行ありチェックが付いていない場合、集合場所、出発時間、使用車両の入力欄は表示しない。
- 同行ありチェックを付けた場合に限り、集合場所、出発時間、使用車両の3項目を動的に表示する。実装方式は条件付きフィールドまたはトグル表示とし、具体的な方式は実装時に決定する。
- 同行ありチェックを外した場合、3項目を再び非表示にし、入力・必須判定の対象外とする。

### 11. 任意項目

MVPで任意項目として用意するもの:

- 出庫要否
- 備考

出庫要否を任意にする理由:

- 先に出庫して用意されている場合がある。
- 必須ではないが、分かると当日の準備がしやすい。

備考にまとめる内容:

- 搬入口
- 受付方法
- 入り方
- 現地で連絡する相手
- 警備員への受付
- その他注意事項

これらは独立項目にしない。現場で対応できる情報であり、項目を増やしすぎると入力負荷が上がるため。

備考欄の上には、入力例として `例: 現地担当者への連絡、注意事項等あれば` のような補助テキストを表示する。

### 12. 現場住所

現場住所は自由入力の大きめのテキスト欄とする。

採用方針:

- 郵便番号、都道府県、市区町村などには分割しない。
- ビル名、階数、部屋番号なども現場住所欄に入力してよい。
- 搬入口、受付方法、現地担当者への連絡、注意事項などは備考欄に入力する。

### 13. 現場到着希望時間

現場到着希望時間は自由入力とする。

採用方針:

- `10:00`、`13:30` のような時刻だけでなく、`午前中`、`13時頃`、`なるべく早め`、`時間指定なし` のような表現も入力できる。
- 現場や顧客との約束が厳密な時刻とは限らないため、時刻入力だけに限定しない。
- 必須項目として扱うが、形式は厳密に制限しない。

採用しない案:

- 時刻選択だけに限定する
- 30分単位の時刻入力に限定する

これらは開始時間・終了時間とは用途が異なるため、現場到着希望時間には適用しない。

### 14. ログイン・権限管理

MVPではログイン処理や個人識別を行わない。

採用方針:

- 誰もが同じ権限で参照・編集できる掲示板のようなシステムにする。
- 社員、配送・設置担当者、管理者のような権限差は設けない。
- ログイン、ユーザー管理、権限管理はMVPに含めない。

理由:

- 現行Excelも全員が同じように編集できる運用である。
- ログインや権限管理を入れると、MVPの操作や説明がややこしくなる。
- 今回の主目的は、Excelに近い一覧性を残しながら詳細情報を入力できるようにすることである。

### 15. 編集権限

MVPでは全員が同じ権限なので、配送・設置担当者も案件を編集できる。

ただし、実運用上は配送・設置担当者が積極的に編集する理由は少ない。受託者である担当者が内容を理解できればよく、不明点があれば依頼者へ連絡する。

多少のヒューマンエラーは許容する。ここは現行Excelと同じ軽い運用を優先し、細かい権限や承認機能は入れない。

### 16. 依頼者名の入力方式

MVPでは依頼者名は自由入力とする。

採用理由:

- 現行Excelに近く、入力が軽い。
- ログインや社員マスタを導入しない方針と整合する。
- 表記ゆれは起こり得るが、MVPでは許容する。

採用しない案:

- 依頼者名の自動入力
- 社員マスタからの候補選択

これらは、ログインや利用者識別が必要になった場合の将来拡張として扱う。

### 17. 自動保存

MVPでは、入力欄からフォーカスが外れたタイミングで自動保存する。

採用方針:

- 入力欄から離れたタイミングで保存する。
- 前のページに戻る操作をした場合も、未保存の入力内容を保存する。
- 確定ボタンは置かない。

採用理由:

- 文字入力のたびに保存する方式より実装がシンプル。
- 入力途中の中途半端な値を保存しにくい。
- 保存忘れを防ぎつつ、現行Excelに近い軽い操作感を保てる。

採用しない案:

- 文字を入力するたびに保存する方式。

この方式は保存漏れを減らせる一方で、DBアクセス回数が増え、入力途中の値が保存されやすくなるため、MVPでは採用しない。

### 18. 対応端末

MVPではPCでの入力を主対象とする。

採用方針:

- 社員の多くはPCからスケジュールを入力する想定。
- PCではスケジュール一覧の閲覧、新規入力、編集、キャンセルを一通り行えるようにする。
- 配送・設置担当者がiPhoneでスケジュール一覧と案件詳細を正常に確認できることを前提にする。
- iPhoneでも基本操作は可能にするが、入力作業の快適さはPCを優先する。
- 月間スケジュール表は横幅が広くなりやすいため、iPhoneでは横スクロールまたは表示調整を前提にする。
- iPhone専用の「今日の案件」「選択日の案件」などの簡易表示はMVPでは作らない。
- iPhoneでは月間表をスクロール、拡大して確認できることを前提にする。

補足:

- PCとiPhoneで同じWebアプリを表示することは可能。
- ただし、画面幅、タップ操作、入力補助の違いがあるため、完全に同じ見え方ではなく、同じ機能を端末幅に合わせて使える状態を目指す。

### 19. 月切り替え

月タブは、時刻同期から得た現在月を基準に、前月・当月・翌月のみ表示する。

採用方針:

- サイトを開いた時点の年月を自動判定し、該当月のスケジュール一覧を最初に表示する。
- 月タブには前月、当月、翌月の3つを表示する。
- それ以外の月を参照したい場合は、スケジュール一覧画面の左上端に置く年月選択の目次から年度と月を選ぶ。
- 年月選択は、過去月や遠い未来月を参照するための補助操作として扱う。
- 普段の運用では、当月とその前後の月へ素早く移動できればよい。

採用しない案:

- 年度内のすべての月を常に月タブとして並べる
- サイトを開くたびに年度選択、月選択、スケジュール一覧の順に遷移させる

これらは普段使いの操作が重くなりやすいため、MVPでは採用しない。

### 20. サンプルデータ

ポートフォリオ用のサンプルデータは、実在情報を使わず、架空の案件として用意する。

採用方針:

- 1日あたり3件程度の案件が入っている状態を基本にする。
- 作業種別は、設置、回収、交換、配達を一通り含める。
- 依頼者名、顧客名、現場住所、車両、備考などはすべて架空データにする。
- 現場住所の市区町村は、名古屋市、豊田市、岡崎市、一宮市など、愛知県内の市に限定する。
- ただし、番地、建物名、顧客名、担当者名などは実在情報を使わず、`愛知県名古屋市サンプル区1-2-3 サンプルビル3階` のような架空表現にする。

採用理由:

- 実運用に近い件数感で、スケジュール一覧の色分け、矢印、詳細フォームを説明しやすくする。
- 作業種別を一通り含めることで、MVPの入力フォームが作業種別ごとに切り替わらない設計を説明しやすくする。
- 愛知県内の市名に限定することで、業務背景に近いデモにしつつ、実在の個人情報や顧客情報を避ける。

### 21. 初期MVPの完成条件

最初の完成地点は、細かなUIを作り込む前に、最小限の項目入力から一覧反映まで実際に動く段階とする。「骨組み」は表示だけのモックを意味しない。

初期MVPに含めるもの:

- 月間スケジュール一覧
- 合意した最小限の項目を持つ案件入力フォーム
- 入力欄から離れたタイミングでの自動保存
- 依頼者名、開始時間、終了時間がそろった案件の一覧反映
- 同じ日の既存案件との時間範囲重複チェックと注意表示
- 二重確認付きの依頼キャンセルと物理削除
- サイト起動時の現在月自動表示
- 現在月を基準とした前月・当月・翌月の月タブ表示と切り替え
- iPhoneでの一覧・入力フォームの表示確認

初期MVPの完成条件に含めないもの:

- 左上の年月選択
- サンプルデータ投入
- 細かなUI調整

その他の合意済み業務処理は破棄せず、初期MVPに含めるかを個別に確認する。

## 上書き・破棄された案

会話途中で出たが、最新方針では採用しない案を明示する。

| 古い案 | 最新方針 |
| --- | --- |
| 通知をMVPの重要機能として扱う | 通知は優先度低め。MVP必須ではない |
| 確定ボタンを置く | 確定ボタンは置かない |
| 入力した時点で確定ステータスにする | ステータス管理自体をMVPでは持たない |
| 確認済みステータスを持つ | 持たない。後から編集しづらくなるため |
| 完了済みステータスを持つ | 持たない。案件数が少なく不要 |
| キャンセル済みステータスを持つ | 持たない。キャンセル時は一覧から表示を消す |
| 依頼者名だけで一覧反映する | 依頼者名、開始時間、終了時間がそろった場合に反映する |
