# Codex / agy CLI 運用ルール

## 目的

このドキュメントは、schedule-system 開発における Codex と agy CLI の役割分担を定義する。

現在の運用では、デービス、ガルシアなどの固定サブエージェント名を使わない。メインの開発判断と実装は Codex が担当し、必要な場合だけ agy CLI をレビュー補助として使う。

## 現在の役割分担

### Codex

Codex は、ユーザーとの壁打ち、方針整理、実装、差分確認、ローカル状態確認を担当する。

Codex が通常担当する作業:

- ユーザー要望の整理
- 変更方針の提案
- 作業ブランチ作成
- 実装
- 差分確認
- 影響範囲確認
- 必要なテストコマンドの提示
- ユーザーが実行する Git コマンドの提示

Codex が原則として行わない作業:

- `main` への直接コミット
- `main` への直接 push
- ユーザーの明示依頼なしの commit
- ユーザーの明示依頼なしの push
- ユーザーの明示依頼なしの PR 作成
- PR の自動マージ
- 本番環境や本番DBの変更操作

### ユーザー

ユーザーは、VS Code 上の作業リポジトリで、手動確認、テスト実行、commit、push、PR作成、PRマージを担当する。

ユーザーが通常担当する作業:

- 画面の手動確認
- 必要なテストコマンドの実行
- `git add`
- `git commit`
- `git push`
- `gh pr create`
- PR 確認
- main へのマージ
- Render の再デプロイ確認

### agy CLI

agy CLI はレビュー補助として使う。実装担当ではなく、計画、差分、テスト観点、リスクの確認に限定する。

agy CLI を使う場面:

- UI / UX の変更範囲が広い場合
- DB、認証、デプロイ、クラウド設定に関係する場合
- 複数ファイルへ影響する実装の場合
- テスト観点の抜け漏れを確認したい場合
- PR 前に第三者視点のリスク確認をしたい場合

agy CLI に依頼しない作業:

- 本番コードの主実装
- commit
- push
- PR 作成
- PR マージ
- 本番環境への接続
- 本番DBへの接続
- 本番認証情報の閲覧

agy CLI の指摘は最終判断ではない。採用可否は Codex が判断し、必要に応じてユーザーへ確認する。

## 作業開始前に読む文書

Codex と agy CLI は、作業前に次を優先して確認する。

1. `AGENTS.md`
2. `引き継ぎ.md`
3. 変更対象に直接関係するコード
4. 変更対象に直接関係するテスト
5. 必要に応じて関連ドキュメント

`AGENTS.md` は恒久的な作業ルール、`引き継ぎ.md` は現在の状況と次作業を管理する文書として扱う。

## agy CLI へ渡すプロンプトの基本形

```text
まず AGENTS.md と 引き継ぎ.md を読んでください。

あなたは schedule-system のレビュー補助エージェントです。
実装、commit、push、PR作成、マージは行わないでください。
本番環境、本番DB、本番認証情報へ接続しないでください。

以下の観点でレビューしてください。

1. 要件逸脱
2. 既存仕様との矛盾
3. データ破損リスク
4. 画面操作上の違和感
5. テスト不足
6. 不要な差分
7. PR分割の妥当性

レビュー対象:
<ここに対象ファイル、差分、計画書、スクリーンショットなどを貼る>

出力形式:
- 採用必須の指摘
- 条件付きで採用すべき指摘
- 採用不要の指摘
- 追加で確認すべき質問
```

## PR前チェック

PR前には、必要に応じて次を確認する。

1. `git status --short --branch`
2. `git diff --stat`
3. 対象ファイルの差分
4. 目的外の変更が混ざっていないこと
5. 必要なテストまたは手動確認
6. agy CLI レビューが必要な変更かどうか

軽微なCSS修正や文言修正では、agy CLI レビューは不要とする。

## 運用上の注意

- 複数の作業フォルダを並行して使わない
- ユーザーがVS Codeで開いているリポジトリを正とする
- 古いCodex作業フォルダと新しいOneDrive作業フォルダを混同しない
- 作業場所が不明な場合は、実装前に停止して確認する
- 外部AIの成果物は、そのまま採用せず、必要な要素だけを理解して再実装する
