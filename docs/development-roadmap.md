# 開発ロードマップ

## 開発方針

最初は上流工程寄りに、現行業務の課題、要件、業務フロー、画面、DB設計を整理する。その後、アジャイル開発とテスト駆動開発を部分的に組み合わせ、最低限のMVPから段階的に機能を増やす。

ポートフォリオとしては、単に画面を作るだけでなく、次の流れを説明できる状態を目指す。

1. 現行業務の課題整理
2. 要件定義
3. 業務フロー設計
4. 画面一覧と画面遷移
5. DB設計
6. MVP実装
7. テスト
8. 振り返りと改善

## ブランチ運用

- mainブランチへ直接変更しない
- 作業単位でブランチを作成する
- Pull Requestで差分を確認してからmainへ取り込む
- ドキュメント、実装、テスト追加は可能な限り別PRに分ける

ブランチ名の例:

- docs/initial-planning
- feat/schedule-list
- feat/job-case-crud
- test/domain-rules
- refactor/service-layer

## フェーズ0: 上流工程整理

目的:

- 実装前に作るものの範囲を明確にする
- MVPと将来拡張を分ける
- SIer就活で説明しやすい資料を整える

成果物:

- README
- 要件定義
- 業務フロー
- 画面一覧
- DB設計
- 開発ロードマップ
- テスト方針

## フェーズ1: 技術検証とプロジェクト土台

目的:

- Java / Spring Bootの最小構成を作る
- DB接続、起動確認、テスト実行の土台を整える
- MVPではログイン、権限管理、ステータス管理を入れない前提を確認する

想定タスク:

- Spring Bootプロジェクト作成
- パッケージ構成決定
- 開発用DB設定
- Lintやフォーマット方針の検討
- GitHub Actionsの初期設定
- READMEへ起動手順を追加

完了条件:

- ローカルでアプリが起動できる
- テストコマンドが実行できる
- CIでビルドとテストが実行できる

## フェーズ2: ドメインモデルとDB実装

目的:

- 案件、時間範囲、同行情報、備考などの基本データを扱えるようにする
- 重要な業務ルールをテストしながら実装する

想定タスク:

- EntityまたはModel定義
- Repository定義
- マイグレーション導入検討
- 案件日時のバリデーション
- 一覧反映条件のルール
- 時間範囲重複チェック
- 同行ありの場合の条件付き必須ルール

TDD対象:

- 終了日時が開始日時より後であること
- 同じ日の時間範囲が既存案件と重ならないこと
- 依頼者名、開始時間、終了時間がそろうまで一覧に反映しないこと
- 同行ありの場合に集合場所、出発時間、使用車両が必須になること

## フェーズ3: 案件CRUD

目的:

- 全利用者が案件を入力・編集・確認できるようにする

想定タスク:

- 案件入力・編集フォーム
- 依頼キャンセル確認ダイアログ
- 入力バリデーション
- エラーメッセージ表示
- 前のページに戻る操作

完了条件:

- 架空データで案件を登録できる
- 入力した案件をフォームで確認・編集できる
- 不正な入力は保存されない
- 依頼キャンセル時に二重確認が表示される

## フェーズ4: スケジュール一覧

目的:

- Excelの代替となる日付・30分単位の一覧を作る

想定タスク:

- 現在月を自動表示する月間一覧画面
- 月タブまたは前月・翌月操作による月切り替え
- セルクリックによる入力・編集フォームへの遷移
- 30分セルへの時間範囲展開
- 先頭セルに依頼者名、2セル目以降に矢印を表示するセル描画
- 1日内の案件順に5色程度から色を割り当てる表示
- 時間範囲重複時の注意表示

完了条件:

- 現在月の案件を時間帯順に確認できる
- 月タブから別月の一覧へ移動できる
- 案件をクリックして詳細へ移動できる
- 同じ日の重複時間帯に別案件を入力できない

## フェーズ5: 担当者向け確認フロー

目的:

- 配送・設置担当者がスケジュール一覧から案件詳細を確認しやすくする

想定タスク:

- 必要情報の表示整理
- 備考欄の見やすさ調整
- 一覧セルに表示する情報の整理

完了条件:

- 担当者が当日の案件を確認できる
- 案件詳細に依頼内容、集合情報、車両、備考が表示される
- 不明点がある場合に依頼者へ連絡すべき内容を把握できる

## フェーズ6: UI調整と入力負荷の削減

目的:

- 現場で使い続けられるよう、入力項目と表示情報を絞り込む

想定タスク:

- 入力フォームの項目順調整
- 必須項目と任意項目の見直し
- 一覧セルの表示内容調整
- 色分けルールの検討
- スマートフォン幅での確認

完了条件:

- 依頼者が迷わず入力できる
- 配送・設置担当者が必要情報をすぐ確認できる
- ログインや権限管理なしでも運用イメージを説明できる

## フェーズ7: 品質向上とデプロイ準備

目的:

- ポートフォリオとして見せられる品質に整える

想定タスク:

- 自動テスト拡充
- エラーハンドリング整理
- READMEのセットアップ手順更新
- サンプルデータ整備
- 画面キャプチャ追加
- デプロイ先検討

完了条件:

- 初見でもREADMEから起動できる
- 主要機能のテストが通る
- 架空データでデモできる

## 将来拡張ロードマップ

| 優先度 | 機能 | 目的 |
| --- | --- | --- |
| 高 | 通知機能 | 案件変更時の見落としを減らす |
| 高 | 変更履歴 | 誰が何を変更したか追跡する |
| 中 | ログイン・権限管理 | 利用者識別が必要になった場合に導入する |
| 中 | Excel取込 | 既存運用から移行しやすくする |
| 中 | CSV出力 | 外部共有や集計に使う |
| 中 | 地図連携 | 作業場所と移動を確認しやすくする |
| 低 | ルート最適化 | 配送順の検討を支援する |
| 低 | 写真アップロード | 作業完了報告を強化する |

## スプリント運用の初期案

- 1スプリントは1週間を目安にする
- 各スプリントで「設計確認」「実装」「テスト」「振り返り」を行う
- 1PRは説明しやすい単位に分ける
- 完了条件をIssueまたはPR本文に明記する

## ポートフォリオで説明する観点

- 現場課題から要件へ落とし込んだ流れ
- Excelの利点を残しつつWeb化した設計判断
- MVPと将来拡張を分けたスコープ管理
- Java / Spring Bootで層を分けた実装方針
- 自動テストで業務ルールを守る方針
- PR運用による開発プロセス
