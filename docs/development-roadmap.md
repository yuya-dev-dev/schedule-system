# 開発ロードマップ

## 開発方針

現行業務の課題、要件、業務フロー、画面、DB設計を整理した後、動作する小さな単位で実装とテストを繰り返す。初期MVPを一度に作り切るのではなく、一覧と入力がつながる最小の縦切りから段階的に完成させる。

ポートフォリオでは、次の流れを説明できる状態を目指す。

1. 現行業務の課題整理
2. 顧客ヒアリングと要件定義
3. 業務フロー、画面、DBの初期設計
4. MVP範囲とリスクのレビュー
5. 小さな機能単位での実装とテスト
6. 利用者による試運転と改善

## ブランチ運用

- mainブランチへ直接変更しない
- 作業単位でブランチを作成する
- Pull Requestで差分を確認してからmainへ取り込む
- ドキュメント、実装、テスト追加は可能な限り別PRに分ける

ブランチ名の例:

- docs/phase0-review-fixes
- chore/spring-boot-setup
- feat/schedule-list
- feat/request-drafts
- test/domain-rules

## フェーズ0: 上流工程整理

目的:

- 実装前に作るものの範囲を明確にする
- MVPとMVP後の機能を分ける
- 文書間の矛盾と実装リスクをレビューする

成果物:

- README
- 要件定義
- 要件ヒアリング議事録
- 業務フロー
- 画面一覧
- DB設計
- 開発ロードマップ
- テスト方針

完了条件:

- 改訂版MVPについて顧客とベンダの認識が一致している
- 要件、業務フロー、画面、DB、テスト方針に重大な矛盾がない
- 実装前に確認すべきリスクが明示されている

## フェーズ1: 技術検証とプロジェクト土台

目的:

- Java / Spring Bootの最小構成を作る
- DB接続、起動確認、自動テストの土台を整える

想定タスク:

- Spring Bootプロジェクト作成
- パッケージ構成決定
- 開発用DB設定
- FlywayなどのDBマイグレーション方針決定
- フォーマット方針決定
- GitHub Actionsの初期設定
- READMEへ起動・テスト手順を追加

完了条件:

- ローカルでアプリが起動できる
- DBへ接続できる
- テストコマンドを実行できる
- CIでビルドとテストが成功する

## フェーズ2: 最小の縦切り機能

目的:

- 月間一覧、入力フォーム、DB保存を最短経路で接続する
- 早い段階でPCとiPhoneから画面を確認できるようにする

想定タスク:

- 日本時間の現在月を表示する月間一覧
- 水曜日・金曜日、8:30-17:30、30分単位のセル生成
- 前月・当月・翌月タブ
- 空白セルから日付を引き継ぐ新規入力フォーム
- `schedule_requests` のEntity、Repository、Service
- 依頼者名、開始時間、終了時間の保存
- 3項目がそろった案件の一覧反映
- 入力済みセルから既存案件フォームへの遷移

完了条件:

- URLを開くと現在月が表示される
- 空白セルから案件を入力してDBへ保存できる
- 保存した案件が正しい30分セルへ表示される
- 入力済みセルから同じ案件を開ける

## フェーズ3: 改訂版MVPの完成

目的:

- 合意済み入力項目と重要な業務ルールを追加し、試運転可能なMVPを完成させる

想定タスク:

- 6種類の作業種別と合意済み入力項目
- 同行あり時の動的項目表示と条件付き必須判定
- 同行解除時の集合場所、出発時間、使用車両の削除
- 出庫状態3種類の入力
- フォーカス離脱時と戻る操作時の自動保存
- 月間一覧上部の下書き一覧、再開、削除
- 作業日を過ぎた下書きの自動物理削除
- 下書きから通常案件へ移る際の重複再確認
- 同時登録の先着優先と具体的な競合エラー
- 同一案件更新時の楽観ロック
- 二重確認付きの依頼キャンセルと物理削除
- 過去月の閲覧専用制御
- 依頼者名、作業種別、矢印、案件色、`＊未入力` の表示
- iPhone、社用スマートフォン、ノートPCでの基本表示確認

TDD対象:

- 終了時間が開始時間より後であること
- 時刻が30分単位かつ8:30-17:30内であること
- 3項目がそろうまで下書きとして扱うこと
- 下書きが時間枠を確保しないこと
- 日本時間で作業日を過ぎた下書きだけを自動削除すること
- 通常案件へ移る際に重複を拒否すること
- 同時登録では先着案件だけが保存されること
- 同行あり時に3項目が必須になること
- 同行解除時に3項目の保存値を削除すること
- 入庫、商品管理では共通必須4項目だけで不足警告を出さないこと

完了条件:

- 改訂版MVPの機能要件と主要テストがすべて通る
- PCとiPhoneで一覧、入力、編集、下書き、キャンセルを確認できる
- 初期MVPに含めない機能が混入していない

## フェーズ4: 品質向上とデモ準備

目的:

- ポートフォリオとして説明でき、利用者へ試してもらえる品質に整える

想定タスク:

- 自動テスト拡充
- 主要ブラウザ操作のE2Eテスト
- エラーメッセージと例外処理の整理
- 架空サンプルデータ整備
- READMEのセットアップ手順更新
- 画面キャプチャと構成図追加
- 要件ID、画面、DB、テストのトレーサビリティ整理

完了条件:

- READMEの手順で起動とテストを再現できる
- 架空データだけで主要業務をデモできる
- 重要な業務ルールに自動テストがある

## フェーズ5: MVP後の業務機能

初期MVPの試運転結果と優先度を確認し、機能ごとに別PRで追加する。

推奨順:

1. 左上の年月選択
2. 外部祝日カレンダー連携と祝日列の自動除外
3. 日付単位の休み設定
4. 案件の別日コピーとコピー先選択画面

補足:

- 祝日同期は日付列を一覧生成時に除外し、案件データを自動削除しない
- 休み設定は全員が操作可能だが、実運用では配送・設置担当者本人またはセールス部長が行う
- 各機能は改めて受入条件とデータ削除リスクを確認してから実装する

## フェーズ6: 試験運用と導入判断

目的:

- 機能の正しさと現場での使いやすさを分けて評価する

実施順:

1. 配送・設置担当者とCodexで機能テストを行う
2. 個人iPhone、社用スマートフォン、ノートPCでブラウザ表示と操作を確認する
3. セールス部門の数名が入力と確認を試し、UIとUXを評価する
4. 指摘を修正し、再確認する
5. セールス部長へ導入を提案し、承認後に実業務で利用する

完了条件:

- 機能テストで重大な不具合が残っていない
- 対象端末で主要操作を完了できる
- 試運転利用者から業務利用可能との評価を得る
- セールス部長が導入を承認する

## 将来検討

| 機能 | 方針 |
| --- | --- |
| 通知 | 試運転で一覧の見落としが発生した場合だけ検討する |
| 変更履歴 | 誤操作や追跡の必要性が高まった場合に検討する |
| ログイン・権限管理 | 会社の情報管理方針または運用上必要になった場合に検討する |
| CSV出力 | 外部共有や集計が必要になった場合に検討する |
| バックアップ・復旧 | 基本機能の完成後、必要に応じて追加する |
| 地図・ルート最適化 | 配送順の支援が必要になった場合に検討する |
| PWA | ブラウザのショートカットでは不足する場合だけ検討する |

## スプリント運用

- 1スプリントは1週間を目安にする
- 各スプリントで設計確認、実装、テスト、振り返りを行う
- 1PRは説明しやすい機能単位に分ける
- 完了条件をIssueまたはPR本文へ明記する

## ポートフォリオで説明する観点

- 現場課題から要件へ落とし込んだ流れ
- レビューでMVPを縮小し、データ消失リスクを抑えた判断
- Excelの一覧性を残しながら詳細情報を構造化した設計
- Java / Spring Bootで層を分けた実装方針
- 自動保存、下書き、同時登録をテストで守る方針
- PR運用と試運転による改善プロセス
