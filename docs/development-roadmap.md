# 開発ロードマップ

## 開発方針

現行業務の課題、要件、業務フロー、画面、DB設計を整理した後、動作する小さな単位で実装とテストを繰り返す。初期MVPを一度に作り切るのではなく、一覧と入力がつながる最小の縦切りから段階的に完成させる。

ポートフォリオでは、次の流れを説明できる状態を目指す。

1. 現行業務の課題整理
2. 顧客ヒアリングと要件定義
3. 業務フロー、画面、DBの初期設計
4. MVP範囲とリスクのレビュー
5. 小さな機能単位での実装とテスト
6. 利用者による試運転と改善

## ブランチ運用

- mainブランチへ直接変更しない
- 作業単位でブランチを作成する
- Pull Requestで差分を確認してからmainへ取り込む
- ドキュメント単独の変更は専用PRに分ける
- 機能実装と、その機能を守る単体・結合テストは原則として同じPRに含める
- 未テストの機能実装だけをmainへ取り込まない

ブランチ名の例:

- docs/phase0-review-fixes
- chore/spring-boot-setup
- feat/schedule-list
- feat/request-drafts
- test/domain-rules

成果物:

- 業務フロー
- 画面一覧
- DB設計

- 要件、業務フロー、画面、DB、テスト方針に重大な矛盾がない
- 実装前に確認すべきリスクが明示されている

## フェーズ1: 技術検証とプロジェクト土台

目的:

- Java / Spring Bootの最小構成を作る
- DB接続、起動確認、自動テストの土台を整える

想定タスク:

- Spring Bootプロジェクト作成
- パッケージ構成決定
- 開発用DB設定
- 本番想定DBを決定し、H2との差異を踏まえた利用範囲を決める
- 本番相当DBで `DRAFT / PUBLISHED`、時間重複、楽観ロックの小さな同時実行検証を行う
- Thymeleafなどの画面方式と、自動保存の通信方式を決定する
- トランザクション分離レベル、ロック方式、競合エラーの返却方式を決定する
- FlywayなどのDBマイグレーション方針決定
- フォーマット方針決定
- GitHub Actionsの初期設定
- READMEへ起動・テスト手順を追加

完了条件:

- ローカルでアプリが起動できる
- DBへ接続できる
- テストコマンドを実行できる
- CIでビルドとテストが成功する
- 画面方式、自動保存通信、採用DB、同時実行方式の技術判断が記録されている
- 本番相当DBで先着案件だけが `PUBLISHED` になる競合テストが通る
- フェーズ2以降の期間目安を、技術検証結果に基づいて暫定設定できる

## フェーズ2: 最小の縦切り機能

目的:

- 月間一覧、入力フォーム、DB保存を最短経路で接続する
- 早い段階でPCとiPhoneから画面を確認できるようにする

想定タスク:

- 日本時間の現在月を表示する月間一覧
- 水曜日・金曜日、8:30-17:30、30分単位のセル生成
- 前月・当月・翌月タブ
- 空白セルから日付を引き継ぐ新規入力フォーム
- `schedule_requests` のEntity、Repository、Service
- 時間範囲、作業種別、依頼者名の保存
- 作業種別ごとの一覧反映条件と `DRAFT / PUBLISHED` の技術状態
- 入力済みセルから既存案件フォームへの遷移
- 空白セルから入力、保存、一覧反映、再表示までを通すスモークE2Eテスト1本

完了条件:

- URLを開くと現在月が表示される
- 空白セルから案件を入力してDBへ保存できる
- 保存した案件が正しい30分セルへ表示される
- 入力済みセルから同じ案件を開ける
- 最小のスモークE2Eテストが安定して通る

フェーズ2実装結果（2026年6月20日）:

- 日本時間の現在月と前月・翌月タブを実装
- 水曜日・金曜日、8:30-17:30、30分単位の月間一覧を実装
- 空白セルからの日付引き継ぎ、新規保存、一覧反映を実装
- 入力済みセルからの確認・編集を実装
- 依頼者名、作業種別、案件色、継続矢印を実装
- 分単位の時間入力、隣接時間の登録、時間重複エラーを実装
- PC幅とiPhone幅の表示確認を実施
- 月生成の単体テストと、入力から再表示までのスモークテストを追加

フェーズ2の「一覧へ戻る」は保存を兼ねる暫定操作とし、フォーカス離脱時の自動保存、保存状態表示、下書き再開はフェーズ3A・3Bで完成させる。

## フェーズ3: 改訂版MVPの完成

目的:

- 合意済み入力項目と重要な業務ルールを追加し、試運転可能なMVPを完成させる

### 3A: 入力項目・必須判定・自動保存

- 6種類の作業種別と合意済み入力項目
- 同行あり時の動的項目表示、集合場所・出発時間の条件付き必須判定、任意の使用車両指定
- 同行解除時の集合場所、出発時間、使用車両の削除
- 出庫状態3種類の入力
- フォーカス離脱時と戻る操作時の自動保存
- 保存中、保存済み、保存失敗、再試行の表示
- フォーム上部の対象日表示、赤字の必須・エラー表示
- 作業種別変更時の詳細削除確認

完了条件:

- 作業種別ごとの必須・任意・条件付き必須が動作する
- 自動保存に成功・失敗した状態を利用者が判断できる
- 機能実装と対応する単体・結合テストが同じPRで通る

### 3B: 下書き・重複・同時操作・削除

- 月間一覧上部の下書き一覧、再開、削除
- 下書き理由と競合時間帯の表示
- 作業日を過ぎた下書きの自動物理削除
- 下書きから通常案件へ移る際の重複再確認
- 同時登録の先着優先と具体的な競合エラー
- 同一案件更新時の楽観ロック
- 二重確認付きの依頼キャンセルと物理削除

完了条件:

- 下書き、時間重複、同時登録、楽観ロック、物理削除の主要テストが通る
- 先着案件だけが `PUBLISHED` となり、後続入力を理由付き `DRAFT` として保持できる
- 機能実装と対応する単体・結合テストが同じPRで通る

### 3C: 一覧表示・過去日・端末確認

- 過去日の閲覧専用制御
- 依頼者名、作業種別、矢印、案件色、`＊未入力` の表示
- 開始時間順に5色を割り当てる表示制御
- iPhoneでの一覧・案件詳細閲覧、ノートPCでの基本操作確認

完了条件:

- セル表示、色、矢印、`＊未入力`、過去日制御が要件どおり動作する
- PCで主要操作、iPhoneで一覧と案件詳細の閲覧を確認できる
- 機能実装と対応するテストが同じPRで通る

TDD対象:

- 終了時間が開始時間より後であること
- 時刻が30分単位かつ8:30-17:30内であること
- 作業種別ごとの一覧反映条件がそろうまで `DRAFT` とすること
- 時間重複した入力を理由付きの `DRAFT` として保持すること
- 下書きが時間枠を確保しないこと
- 日本時間で作業日を過ぎた下書きだけを自動削除すること
- 通常案件へ移る際に重複を拒否すること
- 同時登録では先着案件だけが `PUBLISHED` となり、後続が理由付き `DRAFT` になること
- 同行あり時に集合場所と出発時間だけが必須になること
- 同行解除時に3項目の保存値を削除すること
- 入庫、商品管理では依頼者名を任意とし、時間範囲と作業種別だけで不足警告を出さないこと

完了条件:

- 改訂版MVPの機能要件と主要テストがすべて通る
- PCで一覧、入力、編集、下書き、キャンセルを確認でき、iPhoneで一覧と案件詳細を閲覧できる
- 初期MVPに含めない機能が混入していない

## フェーズ4: 品質向上とデモ準備

目的:

- ポートフォリオとして説明でき、利用者へ試してもらえる品質に整える

想定タスク:

- 自動テスト拡充
- 主要ブラウザ操作のE2Eテスト
- エラーメッセージと例外処理の整理
- 架空サンプルデータ整備
- READMEのセットアップ手順更新
- 画面キャプチャと構成図追加
- 要件ID、画面、DB、テストのトレーサビリティ整理
- 架空データだけを使う一時共有環境の構築
- PC、iPhone、社用スマートフォンから共有URLへ接続する技術検証
- 複数端末での同時操作、アプリ再起動、DBデータ保持の確認
- 試験運用前に、共有方法が会社の情報管理方針に反しないことを確認する

完了条件:

- READMEの手順で起動とテストを再現できる
- 架空データだけで主要業務をデモできる
- 重要な業務ルールに自動テストがある
- 架空データ限定の共有URLへ対象端末から接続できる
- 共有環境の再起動後もDBデータが保持される
- 社員へ試してもらう前の情報管理確認が完了している

## フェーズ5: MVP試験運用と改善

目的:

- 機能の正しさと現場での使いやすさを分けて評価する
- 追加機能の優先順位を実際の試験結果から決める

実施順:

1. 配送・設置担当者とCodexで機能テストを行う
2. 個人iPhoneと社用スマートフォンで一覧・案件詳細の閲覧を確認し、ノートPCで一連の操作を確認する
3. 許可を得たセールス部門の数名が、架空データで入力と確認を試し、UIとUXを評価する
4. 指摘を機能不具合、操作性改善、追加要望に分類する
5. MVP内の指摘を修正して再確認する
6. セールス部長へ試験結果を説明し、正式導入検討へ進むか判断してもらう

完了条件:

- 機能テストで重大な不具合が残っていない
- 対象端末でMVPの合格条件を確認できる
- 試験運用利用者からUIとUXの評価を収集できる
- MVP後機能の優先順位を判断できる
- 正式導入検討へ進むかの判断結果が記録されている

## フェーズ6: MVP後の業務機能

フェーズ5の試験結果と優先度を確認し、必要な機能だけを機能実装と対応テストを含む別PRで追加する。

初期候補:

1. 左上の年月選択
2. 外部祝日カレンダー連携と祝日列の自動除外
3. 日付単位の休み設定
4. 案件の別日コピーとコピー先選択画面

補足:

- 上記の順序は固定せず、試験運用結果に基づいて決める
- 祝日同期は日付列を一覧生成時に除外し、案件データを自動削除しない
- 休み設定は全員が操作可能だが、実運用では配送・設置担当者本人またはセールス部長が行う
- 各機能は改めて受入条件とデータ削除リスクを確認してから実装する

## フェーズ7: 正式運用準備と導入判断

目的:

- 実在案件を入力する前に、配置、情報管理、バックアップ、復旧方法を確認する

想定タスク:

- 正式なアプリとDBの配置先を決定する
- 会社の情報管理方針と共有方法を確認する
- DBバックアップの取得方法と保管先を決定する
- バックアップからの復元手順を作成し、架空データで復元テストを行う
- 誤削除、障害、連絡先を含む最低限の運用手順を作成する
- セールス部長と必要な社内担当者へ試験結果と運用案を説明する

完了条件:

- 会社が正式配置先と実データ利用を承認している
- バックアップ取得と復元テストが成功している
- 障害・誤削除時の最低限の対応手順がある
- セールス部長が正式導入を承認している
- 上記を満たした後にのみ実在案件の入力を開始する

## 将来検討

| 機能 | 方針 |
| --- | --- |
| 通知 | 試運転で一覧の見落としが発生した場合だけ検討する |

| CSV出力 | 外部共有や集計が必要になった場合に検討する |

| PWA | ブラウザのショートカットでは不足する場合だけ検討する |
| AIアシスタント | MVP完成後、架空データとローカルAIを優先して使い方Q&Aと確認付き入力支援を試作する。会社承認なしに実データを外部送信しない |

## スプリント運用

- 期間は確約日ではなく、完了条件を伴う暫定目安として扱う
- フェーズ1の技術検証が終わった時点で、フェーズ2以降の見積もりを更新する
- 1スプリントは1週間を目安とし、各スプリントに1〜2日の調査・修正バッファを見込む
- 各スプリントで設計確認、実装、テスト、振り返りを行う
- 1PRは説明しやすい機能単位に分ける
- 機能実装と対応する単体・結合テストは同じPRに含める
- 完了条件をIssueまたはPR本文へ明記する

暫定的な進行イメージ:

| 期間目安 | 主な対象 | 終了条件 |
| --- | --- | --- |
| 1週目 | フェーズ1とフェーズ2 | 技術判断を記録し、最小縦切りとスモークE2Eが通る |
| 2週目 | フェーズ3Aと3Cの一覧基礎 | 入力・自動保存・主要一覧表示が動く |
| 3週目 | フェーズ3B | 下書き・競合・同時操作・削除のテストが通る |
| 4週目 | フェーズ3C残作業とフェーズ4 | 端末確認、品質整理、一時共有環境の検証が終わる |
| 5週目以降 | フェーズ5以降 | 試験結果に応じて改善、追加機能、正式運用準備を進める |

この表は学習や技術調査の結果に応じて更新し、日付より各フェーズの完了条件を優先する。

## ポートフォリオで説明する観点

- 現場課題から要件へ落とし込んだ流れ
- レビューでMVPを縮小し、データ消失リスクを抑えた判断

- Java / Spring Bootで層を分けた実装方針
- 自動保存、下書き、同時登録をテストで守る方針
- PR運用と試運転による改善プロセス
