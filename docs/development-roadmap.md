# 開発ロードマップ

## 開発方針

最初は上流工程寄りに、現行業務の課題、要件、業務フロー、画面、DB設計を整理する。その後、アジャイル開発とテスト駆動開発を部分的に組み合わせ、最低限のMVPから段階的に機能を増やす。

ポートフォリオとしては、単に画面を作るだけでなく、次の流れを説明できる状態を目指す。

1. 現行業務の課題整理
2. 要件定義
3. 業務フロー設計
4. 画面一覧と画面遷移
5. DB設計
6. MVP実装
7. テスト
8. 振り返りと改善

## ブランチ運用

- mainブランチへ直接変更しない
- 作業単位でブランチを作成する
- Pull Requestで差分を確認してからmainへ取り込む
- ドキュメント、実装、テスト追加は可能な限り別PRに分ける

ブランチ名の例:

- docs/initial-planning
- feat/schedule-list
- feat/job-case-crud
- test/domain-rules
- refactor/service-layer

## フェーズ0: 上流工程整理

目的:

- 実装前に作るものの範囲を明確にする
- MVPと将来拡張を分ける
- SIer就活で説明しやすい資料を整える

成果物:

- README
- 要件定義
- 業務フロー
- 画面一覧
- DB設計
- 開発ロードマップ
- テスト方針

## フェーズ1: 技術検証とプロジェクト土台

目的:

- Java / Spring Bootの最小構成を作る
- DB接続、起動確認、テスト実行の土台を整える

想定タスク:

- Spring Bootプロジェクト作成
- パッケージ構成決定
- 開発用DB設定
- Lintやフォーマット方針の検討
- GitHub Actionsの初期設定
- READMEへ起動手順を追加

完了条件:

- ローカルでアプリが起動できる
- テストコマンドが実行できる
- CIでビルドとテストが実行できる

## フェーズ2: ドメインモデルとDB実装

目的:

- 案件、担当者、車両、持参物などの基本データを扱えるようにする
- 重要な業務ルールをテストしながら実装する

想定タスク:

- EntityまたはModel定義
- Repository定義
- マイグレーション導入検討
- 案件日時のバリデーション
- 主担当必須ルール
- ステータス遷移ルール

TDD対象:

- 終了日時が開始日時より後であること
- 主担当が1名以上必要であること
- キャンセル済み案件の扱い

## フェーズ3: 案件CRUD

目的:

- 社員が案件を登録・編集・確認できるようにする

想定タスク:

- 案件登録画面
- 案件編集画面
- 案件詳細画面
- 入力バリデーション
- エラーメッセージ表示
- 登録者、更新者の記録

完了条件:

- 架空データで案件を登録できる
- 登録した案件を詳細画面で確認できる
- 不正な入力は保存されない

## フェーズ4: スケジュール一覧

目的:

- Excelの代替となる日付・時間帯別の一覧を作る

想定タスク:

- 週表示または日表示の一覧画面
- 日付移動
- 担当者絞り込み
- ステータス表示
- 案件詳細へのリンク

完了条件:

- 指定日の案件を時間帯順に確認できる
- 案件をクリックして詳細へ移動できる
- 同じ時間帯の複数案件も視認できる

## フェーズ5: 担当者向け確認フロー

目的:

- 配送・設置担当者が自分の案件を確認しやすくする

想定タスク:

- 自分の担当案件画面
- 確認済みステータス
- 完了ステータス
- 必要情報の表示整理

完了条件:

- 担当者が自分の当日案件を確認できる
- 案件詳細に集合情報、車両、持参物が表示される
- 作業後に完了へ更新できる

## フェーズ6: 認証・権限

目的:

- ユーザーごとの操作範囲を分ける

想定タスク:

- ログイン
- ロール別アクセス制御
- 管理者向けユーザー管理
- 管理者向け車両管理

完了条件:

- 管理者、社員、担当者で操作できる範囲が分かれる
- 更新者を記録できる

## フェーズ7: 品質向上とデプロイ準備

目的:

- ポートフォリオとして見せられる品質に整える

想定タスク:

- 自動テスト拡充
- エラーハンドリング整理
- READMEのセットアップ手順更新
- サンプルデータ整備
- 画面キャプチャ追加
- デプロイ先検討

完了条件:

- 初見でもREADMEから起動できる
- 主要機能のテストが通る
- 架空データでデモできる

## 将来拡張ロードマップ

| 優先度 | 機能 | 目的 |
| --- | --- | --- |
| 高 | 通知機能 | 案件変更時の見落としを減らす |
| 高 | 変更履歴 | 誰が何を変更したか追跡する |
| 中 | Excel取込 | 既存運用から移行しやすくする |
| 中 | CSV出力 | 外部共有や集計に使う |
| 中 | 地図連携 | 作業場所と移動を確認しやすくする |
| 低 | ルート最適化 | 配送順の検討を支援する |
| 低 | 写真アップロード | 作業完了報告を強化する |

## スプリント運用の初期案

- 1スプリントは1週間を目安にする
- 各スプリントで「設計確認」「実装」「テスト」「振り返り」を行う
- 1PRは説明しやすい単位に分ける
- 完了条件をIssueまたはPR本文に明記する

## ポートフォリオで説明する観点

- 現場課題から要件へ落とし込んだ流れ
- Excelの利点を残しつつWeb化した設計判断
- MVPと将来拡張を分けたスコープ管理
- Java / Spring Bootで層を分けた実装方針
- 自動テストで業務ルールを守る方針
- PR運用による開発プロセス
