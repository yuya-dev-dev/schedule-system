# フェーズ4 テスト設計

## 1. 目的

初期MVPを社員へ見せる前に、配送・設置担当者とCodexで機能の正しさを可能な限り確認する。本書は、テスト方針を実行可能なテストケースへ具体化し、何を自動化し、何を実機で確認し、どの状態ならフェーズ4を完了できるかを定義する。

本書ではテスト設計だけを扱う。E2Eテストの実装、共有環境の構築、サンプルデータ投入、UI修正は別作業とする。

## 2. 対象範囲

対象はフェーズ3Cまでに完成した初期MVPとする。

- 月間スケジュール一覧
- 前月、当月、翌月の切り替え
- 案件入力、編集、詳細表示
- フォーカス離脱時と一覧へ戻る操作時の自動保存
- `DRAFT / PUBLISHED` の状態遷移
- 下書き一覧、再開、削除、自動削除
- 時間重複、同時登録、楽観ロック
- 同行情報、出庫状態、作業種別ごとの必須判定
- 依頼キャンセルと物理削除
- 過去日の閲覧専用制御
- PC表示とiPhone閲覧

祝日連携、休み設定、案件コピー、任意年月選択、通知、AI支援、正式運用向けバックアップは対象外とする。採用時に本書へケースを追加する。

## 3. 品質目標

優先順位は、発生確率だけでなく、業務影響と復旧困難度を基準に決める。

| 優先度 | 定義 | 代表的な品質目標 |
| --- | --- | --- |
| P0 | データ消失、二重予約、過去データ改変など、客先対応へ直結する | 入力値を失わない、重複案件を公開しない、過去案件を変更しない |
| P1 | 主要業務を完了できない、または誤操作を誘発する | 入力、保存、再開、編集、削除の主要経路を完了できる |
| P2 | 回避策はあるが、日常利用で混乱や負担が生じる | 表示、案内、端末差、エラー内容が業務上理解できる |
| P3 | 軽微な見た目や補助情報の問題 | 文言や余白などを継続改善できる |

フェーズ4ではP0とP1を自動テスト中心で保証し、P2をE2Eと手動端末確認で保証する。

## 5. テストレベルと責務

| レベル | 責務 | 実行環境 | 主な対象 |
| --- | --- | --- | --- |
| ドメイン単体 | 業務ルールを高速に検証 | JUnit 5 | 時刻、必須条件、状態遷移、値の消去 |
| Service結合 | トランザクションと保存結果を検証 | Spring Boot Test + H2 | 自動保存、下書き、編集、削除 |
| PostgreSQL結合 | DB固有の競合を検証 | Testcontainers PostgreSQL 17 | 排他制約、同時登録、楽観ロック |
| Controller/HTML | HTTP、画面遷移、HTML属性を検証 | MockMvc | URL、エラー、disabled、リンク分岐 |
| ブラウザE2E | HTML、CSS、JavaScript、HTTP、DBの接続を検証 | Playwright for Java + 実アプリ | 自動保存、動的フォーム、主要経路 |
| 手動端末 | Safari差、拡大、スクロール、操作感を検証 | 個人iPhone、社用スマートフォン、ノートPC | 一覧、詳細、月タブ、主要操作 |
| 運用・復旧 | プロセス停止後の保持と復旧を検証 | ローカルまたは共有環境 | アプリ再起動、DB再接続、再試行 |

同じルールをすべての層で重複検証しない。ドメイン試験で条件分岐を網羅し、E2Eは利用者が通る代表経路に絞る。PostgreSQL固有の保証はH2で代替しない。

## 6. テスト環境

| 環境ID | 用途 | DB | 時刻 | 備考 |
| --- | --- | --- | --- | --- |
| ENV-UT | 単体・MockMvc・Service試験 | H2インメモリ | 固定Clock | 高速な回帰試験 |
| ENV-PG | DB競合試験 | Testcontainers PostgreSQL 17 | 固定Clock | Docker Desktop必須 |
| ENV-E2E | ブラウザE2E | テスト専用H2 | `e2e` プロファイルの固定Clock | UI、JavaScript、HTTP接続を検証 |
| ENV-PC | PC手動確認 | 共有環境のDB | 日本時間 | ノートPCの標準ブラウザ |
| ENV-IP | iPhone実機確認 | 共有環境のDB | 日本時間 | Safari、スクロールと拡大を含む |

E2EはPlaywright for Javaと `@SpringBootTest(webEnvironment = RANDOM_PORT)` を組み合わせる。`e2e` プロファイルで `2026-06-20 12:00 Asia/Tokyo` の固定Clockを注入し、テストクラスからRepositoryを使って各ケースの前後に架空fixtureを準備・削除する。

通常起動や正式配置で有効になるデータ初期化APIは作らない。E2Eを並列化する場合はワーカーごとにH2のDB名を分け、同じDBを共有しない。UI中心のE2E合格だけでP0の重複保証を完了扱いにせず、PostgreSQL固有の排他制約と同時登録は必ず `ENV-PG` で確認する。

- 依頼内容: コーヒーサーバー一式の設置、既存機器の回収
- 集合場所: 名古屋支店
- 使用車両の指定: 車両A
- 備考: 到着後に架空担当者へ連絡

## 8. 既存自動テストの評価

工程1では既存46件をベースラインとして再実行し、全件成功を確認した。工程2、3のP0/P1追加後は63件となった。主要ブラウザE2E 6件の追加後は合計69件となり、H2、Playwright Chromium、PostgreSQL Testcontainersを含む全件成功を確認している。

| テスト群 | 現在保証している内容 | フェーズ4での扱い |
| --- | --- | --- |
| `ScheduleRequestTest` | 時刻、一覧反映条件、必須項目、同行、内部作業 | 維持し、不足境界だけ追加 |
| `ScheduleRequestAutosaveServiceTest` | 同一下書き更新、重複、STALE、公開済み編集 | P0回帰試験として維持 |
| `ScheduleRequestWorkflowTest` | H2上の下書き・公開遷移 | H2とPostgreSQLの役割差を明記 |
| `PostgreSqlConcurrencyTest` | 同時登録、隣接時間、楽観ロック | P0ゲートとして必須 |
| `MonthScheduleServiceTest` | 月生成、過去日、初期表示対象日 | 月境界ケースを追加 |
| `ScheduleDatePolicyTest` | 日付、曜日、表示月の登録可否 | 月末・年末境界を追加 |
| `ScheduleVerticalSliceTest` | HTTPからDBまでの主要経路 | HTML/Controller回帰として維持 |

主な不足は、実ブラウザJavaScript、通信障害後の復旧、2画面同時操作、アプリ再起動後のデータ保持、実機Safariである。

## 18. テストで固定する状態遷移

確定要件に基づき、新規入力・下書きと公開済み案件の編集を次のように分ける。

- 新規入力または既存下書きが一覧反映条件を満たさない場合は `DRAFT` とし、時間枠を確保しない。
- 公開済みの通常案件で依頼者名、開始時間、終了時間のいずれかが一時的に欠けた場合は変更をDBへ保存せず、元の `PUBLISHED` レコードと時間枠を維持する。画面上の入力値を残し、不足項目を表示する。
- 公開済みの入庫、商品管理で開始時間、終了時間、作業種別のいずれかが一時的に欠けた場合も、元の `PUBLISHED` レコードと時間枠を維持する。
- 有効な一覧反映条件が再びそろい、重複がない場合に初めて公開済みレコードを更新する。
- 詳細必須項目だけが欠けた通常案件は入力済み値を保存し、`PUBLISHED` と時間枠を維持して `＊未入力` を表示する。
- 公開済み案件の時間変更が既存案件と重複した場合は変更を保存せず、元の `PUBLISHED` レコードと時間枠を維持する。

`SAVE-015` では、一覧反映条件の一時不足、詳細不足による `＊未入力`、時間重複の3経路がいずれも元の予約枠を失わないことを検証する。
