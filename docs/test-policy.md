# テスト方針

## 目的

配送・設置案件のスケジュール管理において、日時、時間範囲、同行条件、一覧反映条件などの業務ルールを安定して守る。ポートフォリオとして、実装だけでなく品質保証の考え方も説明できる状態にする。

## 基本方針

- 重要な業務ルールは自動テストで守る
- 画面表示だけに依存せず、ドメインロジックとサービス層を重点的にテストする
- MVPでは単体テストと結合テストを中心にする
- E2Eテストや負荷テストは将来拡張として段階的に追加する
- テストデータには実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない

## テストレベル

| レベル | 対象 | ツール候補 | MVP |
| --- | --- | --- | --- |
| 単体テスト | ドメイン、バリデーション、サービス | JUnit 5, AssertJ | 対象 |
| Repositoryテスト | DBアクセス | Spring Boot Test, H2 | 対象 |
| Controllerテスト | HTTPリクエスト、画面遷移 | MockMvc | 対象 |
| 結合テスト | 主要ユースケース | Spring Boot Test | 一部対象 |
| E2Eテスト | ブラウザ操作 | Playwrightなど | 将来拡張 |
| 負荷テスト | 一覧表示性能 | JMeterなど | 将来拡張 |

## TDDを適用する領域

すべてを厳密なTDDにするのではなく、バグが業務に直結しやすいルールを優先してTDDにする。

対象候補:

- 終了日時は開始日時より後でなければならない
- 同じ日の時間範囲が既存案件と重ならない
- 依頼者名、開始時間、終了時間がそろうまで一覧に反映しない
- 同行ありの場合、集合場所、出発時間、使用車両が必須になる
- 依頼キャンセル時に二重確認を表示する

## 主要テストケース

### 案件登録

| ケース | 期待結果 |
| --- | --- |
| 必須項目を入力する | 案件が保存され、入力・編集フォームで確認できる |
| 入力欄から離れる | 入力内容が自動保存される |
| 前のページに戻る | 未保存内容が保存されてから一覧へ戻る |
| 開始日時より前の終了日時を入力する | エラーとなり保存されない |
| 既存案件と重なる時間範囲を入力する | エラーとなり一覧へ反映されない |
| 依頼者名のみ入力する | 開始時間と終了時間が未入力のため一覧へ反映されない |
| 同行ありで集合場所なしにする | エラーとなる |
| 同行なしで使用車両を空にする | 保存できる |

### 案件編集

| ケース | 期待結果 |
| --- | --- |
| 日時を変更する | 一覧と詳細に変更が反映される |
| 時間範囲を変更する | 対象の30分セルが変わる |
| 依頼キャンセルを実行する | 二重確認後、一覧から表示が消える |

### スケジュール一覧

| ケース | 期待結果 |
| --- | --- |
| サイトを開く | 現在月のスケジュール一覧が表示される |
| 月タブを押す | 対象月のスケジュール一覧へ切り替わる |
| 指定日の案件を表示する | 時間帯順に表示される |
| 空白セルをクリックする | その日付の新規入力フォームが開く |
| 入力済みセルをクリックする | そのセルに対応する既存案件の入力・編集フォームが開く |
| 30分単位の時間範囲を表示する | 開始時間から終了時間まで同じ色で表示される |
| 複数セルにまたがる案件を表示する | 先頭セルに依頼者名、2セル目以降に矢印が表示される |
| 同じ日に複数案件を表示する | 案件ごとに異なる色が割り当てられる |
| 同時間帯に別案件を入力する | 重複エラーが表示される |

### 担当者向け確認

| ケース | 期待結果 |
| --- | --- |
| 担当者がスケジュール一覧を開く | 当日の案件を確認できる |
| 案件詳細を開く | 依頼内容、集合情報、車両、備考が表示される |

## テストデータ方針

テストデータは架空情報のみ使用する。

| 種別 | 例 |
| --- | --- |
| 顧客名 | サンプル株式会社、テスト商事 |
| 担当者 | 社員A、担当者B |
| 住所 | 東京都サンプル区1-2-3 |
| 車両 | 車両A |
| 電話番号 | 000-0000-0000 |

本番業務で使われる実在情報、画像内に含まれる可能性のある実名、住所、会社名、電話番号、車両番号はテストに使わない。

## CI方針

将来的にGitHub Actionsで次を自動実行する。

- ビルド
- 単体テスト
- Repositoryテスト
- Controllerテスト
- フォーマットチェック

MVP初期では、まずローカルで再現可能なテストコマンドをREADMEに記載する。

## 手動テスト観点

自動テストだけでは確認しにくい操作性は、手動で確認する。

- スケジュール一覧から案件詳細へ迷わず移動できるか
- Excelに慣れたユーザーでも日付と時間帯を把握しやすいか
- 案件詳細に当日必要な情報がまとまっているか
- ログインなし、同一権限の掲示板型UIとして迷わず使えるか
- iPhone幅でもスケジュール一覧と案件詳細を正常に確認できるか
- iPhoneでスクロール、拡大しながら月間表を確認できるか
- PC入力を主対象としつつ、iPhoneで確認時に表示崩れが起きないか
- エラー時の表示が分かりやすいか

## 完了条件

MVPの主要機能について、次を満たすことを目標とする。

- 重要な業務ルールに単体テストがある
- 案件登録、編集、一覧表示の主要パスに結合テストがある
- サンプルデータがすべて架空情報である
- READMEの手順でテストを再実行できる
