# テスト方針

## 目的

配送・設置案件のスケジュール管理において、日時、担当者、車両、ステータスなどの業務ルールを安定して守る。ポートフォリオとして、実装だけでなく品質保証の考え方も説明できる状態にする。

## 基本方針

- 重要な業務ルールは自動テストで守る
- 画面表示だけに依存せず、ドメインロジックとサービス層を重点的にテストする
- MVPでは単体テストと結合テストを中心にする
- E2Eテストや負荷テストは将来拡張として段階的に追加する
- テストデータには実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない

## テストレベル

| レベル | 対象 | ツール候補 | MVP |
| --- | --- | --- | --- |
| 単体テスト | ドメイン、バリデーション、サービス | JUnit 5, AssertJ | 対象 |
| Repositoryテスト | DBアクセス | Spring Boot Test, H2 | 対象 |
| Controllerテスト | HTTPリクエスト、画面遷移 | MockMvc | 対象 |
| 結合テスト | 主要ユースケース | Spring Boot Test | 一部対象 |
| E2Eテスト | ブラウザ操作 | Playwrightなど | 将来拡張 |
| 負荷テスト | 一覧表示性能 | JMeterなど | 将来拡張 |

## TDDを適用する領域

すべてを厳密なTDDにするのではなく、バグが業務に直結しやすいルールを優先してTDDにする。

対象候補:

- 終了日時は開始日時より後でなければならない
- 案件には主担当が1名以上必要
- ステータス遷移が不正な順序にならない
- キャンセル済み案件を通常の完了案件として扱わない
- 無効化された車両を新規案件に割り当てない
- 担当者絞り込みで対象外の案件が混ざらない

## 主要テストケース

### 案件登録

| ケース | 期待結果 |
| --- | --- |
| 必須項目を入力して登録する | 案件が保存され、詳細画面で確認できる |
| 開始日時より前の終了日時を入力する | エラーとなり保存されない |
| 主担当なしで登録する | エラーとなり保存されない |
| 車両未定で登録する | 保存できる |
| 無効な車両を選択する | エラーとなり保存されない |

### 案件編集

| ケース | 期待結果 |
| --- | --- |
| 日時を変更する | 一覧と詳細に変更が反映される |
| 担当者を変更する | 自分の担当案件画面の表示対象が変わる |
| ステータスをキャンセルにする | 一覧上でキャンセルとして区別される |
| 更新者が存在する | 最終更新者として記録される |

### スケジュール一覧

| ケース | 期待結果 |
| --- | --- |
| 指定日の案件を表示する | 時間帯順に表示される |
| 担当者で絞り込む | 対象担当者の案件だけ表示される |
| ステータスで絞り込む | 対象ステータスの案件だけ表示される |
| 同時間帯に複数案件がある | 複数案件が判別できる |

### 担当者向け確認

| ケース | 期待結果 |
| --- | --- |
| 担当者がログインする | 自分の担当案件を確認できる |
| 案件詳細を開く | 集合情報、車両、持参物、注意事項が表示される |
| 作業後に完了へ更新する | ステータスが完了になる |

## テストデータ方針

テストデータは架空情報のみ使用する。

| 種別 | 例 |
| --- | --- |
| 顧客名 | サンプル株式会社、テスト商事 |
| 担当者 | 社員A、担当者B |
| 住所 | 東京都サンプル区1-2-3 |
| 車両 | 車両A |
| 電話番号 | 000-0000-0000 |

本番業務で使われる実在情報、画像内に含まれる可能性のある実名、住所、会社名、電話番号、車両番号はテストに使わない。

## CI方針

将来的にGitHub Actionsで次を自動実行する。

- ビルド
- 単体テスト
- Repositoryテスト
- Controllerテスト
- フォーマットチェック

MVP初期では、まずローカルで再現可能なテストコマンドをREADMEに記載する。

## 手動テスト観点

自動テストだけでは確認しにくい操作性は、手動で確認する。

- スケジュール一覧から案件詳細へ迷わず移動できるか
- Excelに慣れたユーザーでも日付と時間帯を把握しやすいか
- 案件詳細に当日必要な情報がまとまっているか
- スマートフォン幅でも最低限確認できるか
- エラー時の表示が分かりやすいか

## 完了条件

MVPの主要機能について、次を満たすことを目標とする。

- 重要な業務ルールに単体テストがある
- 案件登録、編集、一覧表示の主要パスに結合テストがある
- サンプルデータがすべて架空情報である
- READMEの手順でテストを再実行できる
