# テスト方針

## 目的

配送・設置案件のスケジュール管理において、日時、時間範囲、下書き、同行条件、一覧反映条件、同時登録などの業務ルールを安定して守る。ポートフォリオとして、実装だけでなく品質保証の考え方も説明できる状態にする。

## 基本方針

- 重要な業務ルールは自動テストで守る
- 画面表示だけに依存せず、ドメインロジックとService層を重点的にテストする
- 自動保存、動的フォーム、下書き再開などの主要ブラウザ操作は、限定的なE2Eテストでも確認する
- UIとUXの評価は、対象端末での手動確認と利用者の試運転で行う
- テストデータには実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない

## テストレベル

| レベル | 対象 | ツール候補 | MVP |
| --- | --- | --- | --- |
| 単体テスト | ドメイン、バリデーション、Service | JUnit 5, AssertJ | 対象 |
| Repositoryテスト | DBアクセス、楽観ロック | Spring Boot Test | 対象 |
| Controllerテスト | HTTPリクエスト、画面遷移 | MockMvc | 対象 |
| 結合テスト | 保存、下書き、重複判定、キャンセル | Spring Boot Test | 対象 |
| E2Eテスト | 自動保存、動的項目、下書き再開 | Playwrightなど | 主要経路のみ対象 |
| 負荷テスト | 一覧表示性能 | JMeterなど | MVP後 |

## TDDを適用する領域

- 終了時間は開始時間より後であること
- 時刻が30分単位かつ8:30-17:30内であること
- 依頼者名、開始時間、終了時間がそろうまで下書きとして扱うこと
- 下書きは時間枠を確保しないこと
- 通常案件へ移る際に既存案件との重複を拒否すること
- 同時登録では先に保存された案件だけを採用すること
- 同一案件の古いバージョンからの上書きを拒否すること
- 同行ありチェック時、集合場所、出発時間、使用車両が必須になること
- 同行解除時に3項目の保存値を削除すること
- 入庫、商品管理では共通必須4項目だけで不足警告を出さないこと

## 主要テストケース

### 自動保存と下書き

| ケース | 期待結果 |
| --- | --- |
| 何も入力せずフォームを離れる | 空の案件レコードを作らない |
| 最初の入力欄から離れる | 入力値が下書きとして保存される |
| 依頼者名だけ入力する | 一覧へ反映されず、下書き一覧へ表示される |
| 空白文字だけを入力する | 未入力として扱われ、空レコードは作成されない |
| 前のページに戻る | 未保存内容を保存してから一覧へ戻る |
| 下書き一覧を開く | 作業日、入力済み依頼者名、最終更新日時が表示される |
| 複数月に下書きがある | 作業日が当日以降である全月の下書きが最終更新日時の新しい順に表示される |
| 依頼者名が空の下書きを表示する | `依頼者名未入力` と表示される |
| `入力を再開` を押す | 保存済みの入力値を持つフォームが開く |
| 下書き削除を確定する | 下書きレコードが物理削除される |
| 作業日当日に下書き一覧を開く | 対象下書きが残り、入力を再開できる |
| 作業日の翌日に下書き一覧を開く | 対象下書きが自動で物理削除され、一覧に表示されない |
| 通常案件の作業日が過ぎる | 自動削除されず保持される |
| 下書きで開始・終了時間を入力する | 時間枠はまだ確保されない |
| 3項目をそろえる | 重複確認後、通常案件として一覧へ反映される |
| 3項目をそろえた時点で時間が埋まっている | 下書きの値を保持し、競合時間を含むエラーを表示する |

### 案件入力・編集

| ケース | 期待結果 |
| --- | --- |
| 開始時間と同じ、または前の終了時間を入力する | エラーとなり通常案件へ移らない |
| 8:30より前または17:30より後を入力する | エラーとなる |
| 30分単位ではない時刻を入力する | エラーとなる |
| 設置、回収、交換、配達で詳細必須項目が不足する | 一覧の先頭セルに `＊未入力` を表示する |
| 未入力項目がある案件フォームを開く | フォーム上部に不足項目名を表示する |
| 現場到着希望時間に `午前中` を入力する | 自由入力として保存できる |
| 同行ありチェックを付ける | 集合場所、出発時間、使用車両を表示し条件付き必須にする |
| 同行ありチェックを外す | 3項目を非表示にし、保存値を削除する |
| 出庫状態の初期表示 | `未回答` が選択されている |
| 出庫状態を変更する | `出庫が必要` または `出庫済み` を保存できる |
| 入庫、商品管理で共通必須4項目だけ入力する | `＊未入力` と不足警告を表示しない |
| 同じ案件を古い画面から更新する | 楽観ロックエラーとなり、入力値を保持して最新情報の再読込を促す |

### 時間重複と同時登録

| ケース | 期待結果 |
| --- | --- |
| 9:00-11:00の既存案件へ10:00-12:00を登録する | 重複エラーとなる |
| 9:00-11:00の既存案件へ11:00-12:00を登録する | 登録できる |
| 2利用者が同じ空き時間へ同時登録する | 先にコミットした1件だけを保存する |
| 後続の同時登録 | 日付と競合時間帯を含むエラーを表示する |
| `＊未入力` の通常案件と重複する | 時間枠確保済みとして重複エラーになる |

### スケジュール一覧

| ケース | 期待結果 |
| --- | --- |
| 日本時間でサイトを開く | 現在年月の一覧が表示される |
| 月タブを表示する | 前月・当月・翌月の3か月分が表示される |
| 対象月の一覧を表示する | その月の水曜日・金曜日が日付順に表示される |
| 一覧を表示する | 8:30-17:30が30分単位で表示される |
| 空白セルをクリックする | その日付の新規入力フォームが開く |
| 入力済みセルをクリックする | 対応する既存案件フォームが開く |
| 複数セルにまたがる案件を表示する | 先頭セルに依頼者名と作業種別、後続セルに矢印を表示する |
| 同じ日に複数案件を表示する | 前後の案件で同じ色が連続しない |
| 過去月の入力済みセルをクリックする | 閲覧専用で確認できる |
| 過去月へ変更リクエストを送る | 画面操作だけでなくService層でも拒否する |

### キャンセル

| ケース | 期待結果 |
| --- | --- |
| 依頼キャンセルを押す | 対象案件の情報を持つ確認ダイアログを表示する |
| 確認を取り消す | 案件を変更しない |
| キャンセルを確定する | 案件を物理削除し、該当セルを空きへ戻す |

## MVP後に追加するテスト

- 任意年月選択による過去・未来月への移動
- 祝日カレンダー同期、キャッシュ、取得失敗時の動作
- 祝日列を一覧生成時に除外し、案件データを自動削除しないこと
- 休み設定、解除、確認、対象案件の扱い
- 案件コピーとコピー先での重複チェック
- 通知、検索、バックアップなど、採用された将来機能

## テストデータ方針

テストデータは架空情報のみ使用する。設置、回収、交換、配達、入庫、商品管理を一通り含め、住所の市名は名古屋市、豊田市、岡崎市、一宮市などに限定する。番地、建物名、顧客名、担当者名は実在情報を使わない。

## CI方針

GitHub Actionsで次を自動実行する。

- ビルド
- 単体テスト
- Repositoryテスト
- Controllerテスト
- フォーマットチェック

E2Eテストは安定して実行できる環境を整えた後、CIへの追加を検討する。

## 手動テストと試運転

機能の正しさは配送・設置担当者とCodexでテストし、端末差と使いやすさは実機と利用者が確認する。

- 個人iPhoneで一覧、入力、編集、下書き、キャンセルを確認する
- 社用スマートフォンで同じ主要操作を確認する
- ノートPCで主要操作を確認する
- ブラウザの共有URLから利用し、PWAインストールを前提にしない
- セールス部門の数名がUIとUXを評価する
- 指摘修正後、セールス部長が導入可否を判断する

## 初期MVPの完了条件

- サイト起動時に現在月が表示される
- 前月・当月・翌月のタブを切り替えられる
- 水曜日・金曜日、8:30-17:30、30分単位の一覧が表示される
- 入力内容が自動保存され、未完成入力を下書き一覧から再開・削除できる
- 3項目がそろった案件だけが一覧へ反映される
- 重複入力と同時登録を安全に拒否できる
- 同行条件、出庫状態、作業種別ごとの必須判定が動作する
- 二重確認後に案件をキャンセルできる
- 過去月を閲覧専用で確認できる
- 対象3端末で主要画面と操作を確認できる
- READMEの手順でテストを再実行できる
