# 要件定義

## 目的

配送・設置案件を共有Excelで管理している現場を想定し、案件の予定と詳細情報をWebアプリ上で一元管理する。現場担当者が出勤後に複数の社員へ個別確認している情報を、案件詳細画面で確認できる状態にする。

## 現行業務の整理

現行Excelは、日付ごとの列と時間帯ごとの行を使い、案件を表形式で配置している。色分けや矢印により、時間帯や作業の継続を表現している。

ただし、Excelセルに書ける情報量が限られるため、次のような情報が分散しやすい。

- 作業先の住所
- 集合時間と集合場所
- 同行者の有無
- 使用車両
- 依頼内容
- 回収物や設置物の詳細
- 当日の注意事項
- 依頼者や確認先

## 現行業務の課題

- セル内に情報を書き切れず、案件詳細が不足する
- 担当者が社員へ個別にメールや電話で確認する必要がある
- 変更や追記が一覧から分かりにくい
- 同じ時間帯に複数案件が重なった場合、負荷や移動の見通しが立てづらい
- 車両や依頼内容の詳細が見えにくい
- 過去案件の検索や振り返りがしにくい
- Excelの自由入力により、表記ゆれや確認漏れが発生しやすい

## 対象利用者

| 利用者 | 主な目的 |
| --- | --- |
| 社員 | 案件を登録し、担当者、時間帯、作業内容を共有する |
| 配送・設置担当者 | 入力された案件の詳細を確認する |
| 全員共通 | ログインや権限差を設けず、同じ権限で参照・編集する |

## 提供価値

- スケジュールと案件詳細を紐づけ、確認作業を減らす
- 入力項目を構造化し、情報の抜け漏れを減らす
- 住所、到着希望時間、同行情報、車両、備考を確認しやすくする
- ログインや権限管理を入れず、現行Excelに近い軽い運用を保つ
- SIer就活用ポートフォリオとして、要件定義からテストまで説明できる構成にする

## MVPの範囲

MVPでは、Excelの代替として最低限必要なスケジュール管理と案件詳細管理に絞る。

### 機能要件

| ID | 機能 | 内容 |
| --- | --- | --- |
| FR-001 | スケジュール一覧表示 | 現在月を自動表示し、日付と30分単位の時間帯を軸に案件を一覧表示できる |
| FR-002 | 案件入力 | 全利用者が依頼者名、時間範囲、作業種別、依頼内容、住所などを入力できる |
| FR-003 | 案件編集 | 登録済み案件の日時、依頼内容、同行情報、備考などを編集できる |
| FR-004 | 案件詳細表示 | 案件ごとの住所、依頼内容、集合情報、車両、備考を確認できる |
| FR-005 | 同行情報管理 | 一人作業か同行ありかを入力し、同行ありの場合は集合場所と出発時間を入力できる |
| FR-006 | 車両情報管理 | 同行ありの場合に使用車両を入力できる |
| FR-007 | 時間重複チェック | 同じ日の既存案件と時間範囲が重なる入力を無効にできる |
| FR-008 | 検索・絞り込み | 日付、依頼者名、作業種別で案件を絞り込める |
| FR-009 | 依頼キャンセル | 二重確認後に案件データを物理削除し、該当セルを未入力扱いに戻せる |
| FR-010 | 共通編集権限 | ログインなしで、全員が同じ権限で参照・編集できる |
| FR-011 | セル表示制御 | 先頭セルに依頼者名、2セル目以降に矢印、案件単位の色を表示できる |
| FR-012 | セルクリック分岐 | 空白セルは新規入力フォーム、入力済みセルは該当案件の確認・編集フォームへ遷移できる |

### 非機能要件

| ID | 項目 | 内容 |
| --- | --- | --- |
| NFR-001 | 操作性 | Excelに近い時間帯別の見方を残しつつ、案件詳細へすぐ移動できる |
| NFR-002 | 保守性 | Java / Spring Bootの標準的な構成で、層を分けて実装する |
| NFR-003 | 拡張性 | 通知、地図連携、履歴管理などを後から追加しやすい構造にする |
| NFR-004 | 情報管理 | 実在の個人情報や顧客情報をサンプルデータに含めない。MVPではログインを置かないため、個人情報を入れすぎない |
| NFR-005 | 端末対応 | PC入力を主対象とし、iPhoneではスクロール、拡大によりスケジュール一覧と案件詳細を正常に確認できる |
| NFR-006 | 可用性 | ポートフォリオ段階では単一アプリ構成とし、停止時の復旧手順を明確にする |
| NFR-007 | 性能 | 一般的な月次スケジュール件数でストレスなく一覧表示できる |
| NFR-008 | テスト容易性 | ドメインロジックとサービス層を自動テストしやすくする |

## 将来拡張

MVP後に検討する機能は次の通り。

- ログイン、利用者識別、権限管理
- 案件ステータス管理
- 案件変更時のメール、チャット、アプリ通知
- Excelファイルからの案件取込
- ExcelまたはCSVへの出力
- Googleカレンダーなど外部カレンダー連携
- 地図表示、移動時間計算、ルート最適化
- 添付ファイル管理
- 作業完了報告、写真アップロード
- 承認フロー
- 詳細な監査ログ
- スマートフォン向けUIの強化

## 業務ルール初期案

- 案件には開始予定日時と終了予定日時を必須で持たせる
- サイトを開いたときは現在年月の月間スケジュール一覧を表示する
- 月タブまたは前月・翌月操作で別月のスケジュール一覧へ移動できる
- スケジュール一覧への反映は、依頼者名、開始時間、終了時間がそろっている場合に限る
- 入力欄から離れたタイミングで自動保存する
- 前のページに戻る操作をした場合も未保存内容を保存する
- 複数セルにまたがる案件では、先頭セルに依頼者名、2セル目以降に矢印を表示する
- 案件ごとに色を割り当て、同じ案件の時間範囲を同じ色で表示する
- セル内には作業種別や依頼内容を表示せず、詳細は入力・編集フォームで確認する
- 空白セルをクリックした場合は、その日付の新規入力フォームへ遷移する
- 入力済みセルをクリックした場合は、そのセルに対応する既存案件の入力・編集フォームへ遷移する
- 空白セルから新規入力する場合も、クリックした時間帯で開始時間・終了時間を固定しない
- 同じ日の既存案件と時間範囲が重なる入力は無効にする
- 同行ありの場合は、集合場所、出発時間、使用車両を必須にする
- 同行なしの場合、使用車両は必須にしない
- 住所や顧客情報は案件詳細に集約する
- ステータス管理はMVPでは行わない

## 対象外

MVPでは次を対象外とする。

- 給与、勤怠、労務管理
- 実在顧客データの移行
- 外部サービスとの本格連携
- 高度な配車最適化
- 請求、売上、在庫管理
- ログイン、権限管理、ユーザー管理
- 確認済み、完了、キャンセル済みなどのステータス管理
- iPhone専用の今日表示、選択日表示などの簡易表示

## 個人情報・機密情報の扱い

設計書、サンプルデータ、テストデータには、実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない。必要な場合は、次のような架空表現に置き換える。

- 顧客名: サンプル株式会社、テスト商事
- 担当者名: 社員A、担当者B
- 住所: 東京都サンプル区1-2-3
- 車両: 車両A、車両B
- 電話番号: 000-0000-0000

## 成功条件

- 担当者が当日の案件と詳細情報をWeb上で確認できる
- 社員が案件入力時に必要情報を入力できる
- Excelセルに収まらない情報を案件詳細として管理できる
- MVPと将来拡張の境界が説明できる
- 要件、画面、DB、テスト方針が一貫している
