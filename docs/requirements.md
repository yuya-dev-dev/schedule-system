# 要件定義

## 目的

配送・設置案件を共有Excelで管理している現場を想定し、案件の予定と詳細情報をWebアプリ上で一元管理する。現場担当者が出勤後に複数の社員へ個別確認している情報を、案件詳細画面で確認できる状態にする。

## 現行業務の整理

現行Excelは、日付ごとの列と時間帯ごとの行を使い、案件を表形式で配置している。色分けや矢印により、時間帯や作業の継続を表現している。

ただし、Excelセルに書ける情報量が限られるため、次のような情報が分散しやすい。

- 作業先の住所
- 集合時間と集合場所
- 同行者の有無
- 使用車両
- 依頼内容
- 回収物や設置物の詳細
- 当日の注意事項
- 依頼者や確認先

## 現行業務の課題

- セル内に情報を書き切れず、案件詳細が不足する
- 担当者が社員へ個別にメールや電話で確認する必要がある
- 変更や追記が一覧から分かりにくい
- 同じ時間帯に複数案件が重なった場合、負荷や移動の見通しが立てづらい
- 車両や依頼内容の詳細が見えにくい
- 過去案件の検索や振り返りがしにくい
- Excelの自由入力により、表記ゆれや確認漏れが発生しやすい

## 対象利用者

| 利用者 | 主な目的 |
| --- | --- |
| 社員 | 案件を登録し、担当者、時間帯、作業内容を共有する |
| 配送・設置担当者 | 入力された案件の詳細を確認する |
| 全員共通 | ログインや権限差を設けず、同じ権限で参照・編集する |

約10人の利用者が、現行の共有Excelと同様に、全員で1つのスケジュールを共同利用する。MVPではログインや追加のアクセス制限を設けない。各利用者は共有されたURLをPCまたはiPhoneのブラウザで開いて利用し、正式なインストール機能は設けない。社員による試験運用では架空データ限定の一時共有環境を使い、共有方法が会社の情報管理方針に反しないことを事前に確認する。実在案件の入力前には、正式な配置先について改めて承認を得る。

ログイン、利用者識別、権限管理は将来拡張としても予定せず、全利用者が同じ権限で操作する方針を継続する。

## 提供価値

- スケジュールと案件詳細を紐づけ、確認作業を減らす
- 入力項目を構造化し、情報の抜け漏れを減らす
- 住所、顧客先到着希望時間、同行情報、車両、備考を確認しやすくする
- ログインや権限管理を入れず、現行Excelに近い軽い運用を保つ
- SIer就活用ポートフォリオとして、要件定義からテストまで説明できる構成にする

## MVPの範囲

最初の開発では、細かなUIを作り込む前に、入力から一覧反映まで実際に動く最小版を初期MVPの完成地点とする。

現行Excelが持つ月単位の一覧性、日付・曜日の見出し、時間帯別の予定表示、連続時間の色分けと矢印は、最低限の業務機能として引き継ぐ。ただし、Excel上で一定していない時間区切りは引き継がず、8:30-17:30を30分単位で表示する。案件セル内の会社名、機械名、住所なども引き継がず、依頼者名、作業種別、継続矢印、案件色、`＊未入力` を表示する。

- 月間スケジュール一覧を表示できる
- 対象月のすべての水曜日・金曜日を日付列として自動生成し、日付順に表示できる
- 合意した最小限の項目を持つ案件入力フォームを表示できる
- フォームの入力内容を自動保存できる
- 未完成の入力を下書き一覧から再開または削除できる
- 設置、回収、交換、配達では依頼者名と時間範囲、入庫、商品管理では時間範囲と作業種別がそろった案件を月間一覧へ反映できる
- 同じ日の既存案件と時間範囲が重なる入力を無効にし、注意表示を出せる
- 同時登録では先に保存された案件を採用し、後続の利用者へ重複した時間帯を含むエラーを表示できる
- 二重確認後に案件を物理削除し、該当時間帯を未入力へ戻せる
- サイト起動時に日本時間の現在年月を取得し、現在月の一覧を自動表示できる
- 現在月を基準とした前月・当月・翌月のタブを表示し、対象月を切り替えられる
- iPhoneで一覧と案件詳細を正常に表示し、スクロール、拡大して閲覧できる

祝日カレンダー連携、休み設定、案件コピー、左上の年月選択、サンプルデータ投入、細かなUI調整は、初期MVPの完成条件に含めない。

| FR-002 | 案件入力 | 全利用者が時間範囲、作業種別、依頼者名、依頼内容、現場住所などを入力できる |
| FR-003 | 案件編集 | 登録済み案件の日時、依頼内容、同行情報、備考などを編集できる |

| FR-005 | 同行情報管理 | 同行ありチェックを付けた場合のみ、集合場所と出発時間を入力できる |

| FR-008 | 時間重複チェック | 同じ日の既存案件と時間範囲が重なる入力を一覧へ反映せず、同時登録では先着案件だけを `PUBLISHED`、後続を理由付き `DRAFT` にできる |

| FR-012 | セルクリック分岐 | 空白セルは新規入力フォーム、入力済みセルは該当案件の確認・編集フォームへ遷移できる |
| FR-013 | 過去日閲覧制御 | 日本時間の今日より前の日付は閲覧専用とし、今日以降を変更できる |
| FR-014 | 保存状態表示 | 自動保存の保存中、保存済み、保存失敗を表示し、保存失敗時は入力値を保持して再試行できる |

| ID | 項目 | 内容 |
| --- | --- | --- |
| NFR-001 | 操作性 | Excelに近い時間帯別の見方を残しつつ、案件詳細へすぐ移動できる |
| NFR-002 | 保守性 | Java / Spring Bootの標準的な構成で、層を分けて実装する |

| NFR-006 | 可用性 | ポートフォリオ段階では単一アプリ構成とし、バックアップを初期MVPの完成条件には含めない。ただし、実在案件の入力前にバックアップ方法を決定し、架空データで復元テストを行う |
| NFR-007 | 性能 | 一般的な月次スケジュール件数でストレスなく一覧表示できる |
| NFR-008 | テスト容易性 | ドメインロジックとサービス層を自動テストしやすくする |
| NFR-009 | 利用方式 | 端末へのインストールを不要とし、共有URLをブラウザで開いて利用できる |
| NFR-010 | 同時操作 | 同じ日への同時登録は先着順で一覧へ反映し、後続入力を失わず競合内容を明示する |

## 将来拡張

MVP後に検討する機能は次の通り。

- 案件ステータス管理
- 案件変更時のメール、チャット、アプリ通知。試運転で一覧の見落としが実際に発生した場合だけ追加を検討する
- ExcelまたはCSVへの出力
- Googleカレンダーなどへの業務予定連携。日本の祝日取得も初期MVP後に追加する
- 地図表示、移動時間計算、ルート最適化
- 添付ファイル管理
- 作業完了報告、写真アップロード
- 承認フロー
- 詳細な監査ログ
- スマートフォン向けUIの強化
- スケジュールデータのバックアップと障害時の復旧。初期MVP外だが、実在案件の入力前には必ず導入判断の対象とする
- 依頼者名、作業種別などによる検索・絞り込み。実運用で必要になった場合に追加する
- 外部祝日カレンダーによる祝日列の自動除外
- 日付単位の休み設定と、その日に登録済みの案件を扱う確認処理
- 案件入力内容の別日コピーとコピー先選択画面
- 左上の年月選択による任意年月への移動
- PWAなどの正式なインストール機能。明確な必要性が生じた場合のみ検討する
- 生成AIによる使い方Q&Aと案件入力支援。実データを外部AIへ送らず、編集・削除は利用者の確認後に既存の業務ルールを通して実行する

## 業務ルール初期案

- 案件には開始予定日時と終了予定日時を必須で持たせる
- サイトを開いたときは日本時間の現在年月を取得し、その月の月間スケジュール一覧を表示する
- 対象月に含まれるすべての水曜日・金曜日を日付列として自動生成し、日付順に並べる
- 月タブは現在月を基準に前月・当月・翌月の3か月分を表示する
- 日本時間の現在日付より前を過去日と判定し、一覧と案件詳細を閲覧専用にする
- 過去日では、新規入力、編集、キャンセルを禁止する。MVP後に追加するコピーと休み設定も過去日では禁止する
- 今日以降の日付を編集可能とする
- 初期MVPの画面から編集できる未来月は翌月までとする
- 年月選択の追加後は未来月の上限を設けず編集可能とする
- MVPで未来日へ登録できるのは水曜日・金曜日とする。休み・祝日の除外は各機能の追加時に適用する
- スケジュール表の表示時間帯は8:30-17:30固定とし、30分単位で表示する
- 案件の開始時間・終了時間は8:30-17:30内のみ入力できる
- 設置、回収、交換、配達では、依頼者名、開始時間、終了時間がそろい、時間重複がない場合に一覧へ反映する
- 入庫、商品管理では依頼者名を任意とし、開始時間、終了時間、作業種別がそろい、時間重複がない場合に一覧へ反映する
- 設置、回収、交換、配達では、依頼者名、開始時間、終了時間がそろっているが他の必須項目が不足している場合、先頭セルに `＊未入力` と表示する
- 入力・編集フォームでは、未入力の必須項目を赤文字で項目名付き表示する
- `＊未入力` が表示されている案件でも、依頼者名、開始時間、終了時間がそろっている場合は時間枠を確保済みとして扱う
- 入力欄から離れたタイミングで自動保存する
- 前のページに戻る操作をした場合も未保存内容を保存する
- 最初の入力値を保存した時点で内部状態を `DRAFT` とし、入力がないままフォームを離れた場合は空レコードを作らない
- 文字入力項目は前後の空白を除去し、空文字だけの場合は未入力として扱う
- 一覧反映条件を満たさない入力、または時間重複で反映を拒否された入力は `DRAFT` とし、下書き一覧から再開または削除できる
- 下書き理由は `入力不足` または `時間重複` とし、時間重複では競合時間帯も下書き一覧へ表示する
- 下書き一覧は、作業日が当日以降である全月の下書きを最終更新日時の新しい順に表示する
- 日本時間 `Asia/Tokyo` の現在日付を基準に、作業日が前日以前になった下書きは下書き一覧の取得前に自動で物理削除する
- 自動削除された下書きは復元せず、削除履歴も保持しない
- 下書きは時間枠を確保せず、一覧反映条件がそろって `PUBLISHED` へ移る際に重複を再確認する
- 重複した場合は下書きの入力値を保持し、競合した時間帯を含むエラーを表示する
- 作業種別は、設置、回収、交換、配達、入庫、商品管理の6種類とする
- 入庫、商品管理は自動生成せず、利用者が通常の入力フォームから手動登録する
- 入庫、商品管理では依頼者名を必須にせず、開始時間、終了時間、作業種別の3項目だけを必須とする
- 入庫、商品管理では、その他の項目が空欄でも `＊未入力` とフォーム上部の不足警告を表示しない
- 入庫、商品管理でも任意でその他の項目を入力できる
- 複数セルにまたがる案件では、先頭セルに依頼者名と作業種別、2セル目以降に矢印を表示する
- 色は5色の固定パレットから、同じ日の開始時間が早い案件順に自動割り当てする
- キャンセルや時間変更後はその日の色を再計算し、6件以上では前後の案件に同じ色が連続しないよう5色を再利用する
- 同じ案件の時間範囲は同じ色で表示する
- セル内には作業種別を表示するが、依頼内容、会社名、機械名、住所などは表示せず、詳細は入力・編集フォームで確認する
- 作業種別によってフォーム項目は切り替えず、細かい違いは依頼内容にまとめて入力する
- 空白セルをクリックした場合は、その日付の新規入力フォームへ遷移する
- 入力済みセルをクリックした場合は、そのセルに対応する既存案件の入力・編集フォームへ遷移する
- 空白セルから新規入力する場合も、クリックした時間帯で開始時間・終了時間を固定しない
- 同じ日の既存案件と時間範囲が重なる入力は無効にする
- 時間範囲は開始時刻を含み終了時刻を含まないものとして扱い、既存案件が12:00-14:00の場合は14:00開始の案件を登録できる
- 同じ時間帯への同時登録は先にコミットした案件だけを `PUBLISHED` とし、後続は `DRAFT` のまま入力値を保持して競合内容を表示する
- 新規登録の時間重複では入力を `DRAFT` として保持し、下書き一覧にも未反映理由と競合時間帯を表示する
- 既存案件の時間変更が重複した場合は変更を保存せず、DB上の元の時間枠を維持する。画面上の変更値は残して赤字エラーを表示する
- 開始時間と終了時間を順番に変更する途中で一時的に不正な範囲になった場合は保存せず、両方が正常になった時点でまとめて保存する
- 同一案件の同時編集を検知した場合は上書きを拒否し、画面上の入力値を残したまま最新情報の再読込を促す
- 同行ありチェックを付けた場合のみ、集合場所、出発時間、使用車両の指定を動的に表示する
- 設置、回収、交換、配達では、同行ありの場合に集合場所と出発時間を条件付き必須とし、不足時は `＊未入力` と赤文字警告の対象にする。使用車両の指定は任意とする
- 同行ありチェックが付いていない場合、集合場所、出発時間、使用車両はフォーム上に表示せず、入力・必須判定の対象にしない
- 同行ありチェックを外した場合、保存済みの集合場所、出発時間、使用車両を削除する
- 出庫状態は必須ではなく、`未回答`、`出庫が必要`、`出庫済み` の3状態から選択する。初期値は `未回答` とする
- 現場住所や顧客情報は案件詳細に集約する
- 現場住所は自由入力の大きめのテキスト欄とし、郵便番号や都道府県などには分割しない
- 顧客先到着希望時間は自由入力とし、時刻だけでなく `午前中`、`午後かつ17時まで`、`時間指定なし` のような表現も許可する
- 必須項目と条件付き必須項目は、項目名の先頭付近に赤字で `＊必須` と表示する
- 入力エラーはフォーム上部と該当項目付近に赤字で表示する
- フォーム上部に `2026年6月19日（金）の案件入力` のように対象日を明記し、日付はフォーム内で変更できない
- 通常作業から入庫・商品管理へ変更し、詳細が入力済みの場合は削除内容を確認する。承認後は新しい作業種別、依頼者名、時間範囲を残し、依頼内容、現場住所、顧客先到着希望時間、同行関連値、備考を削除して出庫状態を未回答へ戻す
- 自動保存中、保存済み、保存失敗をフォーム上部へ表示する。失敗時は入力値を保持し、赤字のエラーと再試行操作を表示する
- ブラウザを閉じる、再読み込みするなどの未保存離脱警告はMVP後に検討する
- 搬入口、受付方法、現地担当者への連絡、注意事項などは備考欄に入力する
- 備考欄には `例: 現地担当者への連絡、注意事項等あれば` のような補助テキストを表示する
- 確認済み、完了などの案件ステータス管理はMVPでは行わない
- 一覧へ反映済みの通常案件はMVPでは自動削除せず無期限に保持する。作業日を過ぎた下書きだけは例外として自動削除する

## リスクと対応方針

| リスク | MVPでの対応 |
| --- | --- |
| ログインなしのURLへ意図しない利用者がアクセスする | 実運用前に配置先が会社の情報管理方針を満たすことを導入判断者が確認する |
| 自動保存した未完成入力を見失う、または蓄積して見づらくなる | 当日以降の全月を対象とする下書き一覧から再開・削除でき、作業日を過ぎた下書きは自動削除する |
| 同時登録で重複案件が作られる | 保存処理内で重複を再確認し、先着案件だけを `PUBLISHED`、後続入力を理由付き `DRAFT` とする |
| 同一案件を古い画面から上書きする | 楽観ロックで検知し、最新値の上書きを拒否する |
| キャンセルによる誤削除 | 対象案件を表示して二重確認する |
| バックアップがない状態でデータを失う | 初期MVPと架空データによる試験運用ではリスクを受容する。実在案件の入力前にバックアップ方法を決定し、復元テストを必須とする |

## 対象外

MVPでは次を対象外とする。

- 給与、勤怠、労務管理
- 現行Excelに登録済みの予定の取込・移行
- 外部サービスとの本格連携
- 高度な配車最適化
- 請求、売上、在庫管理
- ログイン、権限管理、ユーザー管理
- 確認済み、完了、キャンセル済みなどのステータス管理
- iPhone専用の今日表示、選択日表示などの簡易表示
- 休憩、移動など、案件ではない時間指定予定の登録
- PWAなどの正式なインストール機能

## 個人情報・機密情報の扱い

設計書、サンプルデータ、テストデータには、実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない。必要な場合は、次のような架空表現に置き換える。

- 顧客名: サンプル株式会社、テスト商事
- 担当者名: 社員A、担当者B
- 住所: 愛知県名古屋市サンプル区1-2-3、愛知県豊田市サンプル町4-5
- 車両: 車両A、車両B
- 電話番号: 000-0000-0000

サンプルデータは、1日あたり3件程度の案件を基本にし、作業種別は設置、回収、交換、配達、入庫、商品管理を一通り含める。現場住所の市名は愛知県内に限定するが、番地、建物名、顧客名、担当者名はすべて架空データにする。

## 成功条件

- 担当者が当日の案件と詳細情報をWeb上で確認できる
- 社員が案件入力時に必要情報を入力できる
- 未完成の入力を下書きとして見失わず、再開または削除できる
- 同時登録で重複案件を作らず、後続利用者が競合内容を理解できる
- 自動保存の状態を確認でき、保存失敗時も入力値を失わず再試行できる
- iPhoneで月間一覧と案件詳細をスクロール、拡大して閲覧できる
- Excelセルに収まらない情報を案件詳細として管理できる
- MVPと将来拡張の境界が説明できる
- 要件、画面、DB、テスト方針が一貫している
