# 要件定義

## 目的

配送・設置案件を共有Excelで管理している現場を想定し、案件の予定と詳細情報をWebアプリ上で一元管理する。現場担当者が出勤後に複数の社員へ個別確認している情報を、案件詳細画面で確認できる状態にする。

## 現行業務の整理

現行Excelは、日付ごとの列と時間帯ごとの行を使い、案件を表形式で配置している。色分けや矢印により、時間帯や作業の継続を表現している。

ただし、Excelセルに書ける情報量が限られるため、次のような情報が分散しやすい。

- 作業先の住所
- 集合時間と集合場所
- 同行者の有無
- 使用車両
- 持参物
- 回収物や設置物の詳細
- 当日の注意事項
- 案件登録者や確認先

## 現行業務の課題

- セル内に情報を書き切れず、案件詳細が不足する
- 担当者が社員へ個別にメールや電話で確認する必要がある
- 変更内容や最終更新者が分かりにくい
- 同じ時間帯に複数案件が重なった場合、負荷や移動の見通しが立てづらい
- 車両や持参物の割り当て状況が見えにくい
- 過去案件の検索や振り返りがしにくい
- Excelの自由入力により、表記ゆれや確認漏れが発生しやすい

## 対象ユーザー

| ユーザー | 主な目的 |
| --- | --- |
| 社員 | 案件を登録し、担当者、時間帯、作業内容を共有する |
| 配送・設置担当者 | 自分に割り当てられた案件の詳細を確認する |
| 管理者 | ユーザー、車両、品目などの基本情報を管理する |

## 提供価値

- スケジュールと案件詳細を紐づけ、確認作業を減らす
- 入力項目を構造化し、情報の抜け漏れを減らす
- 担当者、車両、持参物を一覧しやすくする
- 変更履歴や更新者を記録し、業務の透明性を高める
- SIer就活用ポートフォリオとして、要件定義からテストまで説明できる構成にする

## MVPの範囲

MVPでは、Excelの代替として最低限必要なスケジュール管理と案件詳細管理に絞る。

### 機能要件

| ID | 機能 | 内容 |
| --- | --- | --- |
| FR-001 | スケジュール一覧表示 | 日付と時間帯を軸に案件を一覧表示できる |
| FR-002 | 案件登録 | 社員が案件名、日時、担当者、作業区分、場所情報、持参物などを登録できる |
| FR-003 | 案件編集 | 登録済み案件の日時、担当者、詳細情報を編集できる |
| FR-004 | 案件詳細表示 | 案件ごとの住所、集合情報、車両、持参物、注意事項を確認できる |
| FR-005 | 担当者割当 | 案件に主担当と同行者を紐づけられる |
| FR-006 | 車両割当 | 案件に使用予定車両を紐づけられる |
| FR-007 | ステータス管理 | 未着手、確認済み、作業中、完了、キャンセルなどの状態を管理できる |
| FR-008 | 検索・絞り込み | 日付、担当者、ステータスで案件を絞り込める |
| FR-009 | 最低限の認証 | ユーザーごとにログインし、登録者と更新者を記録できる |
| FR-010 | 権限管理 | 管理者、社員、担当者で利用できる操作を分けられる |

### 非機能要件

| ID | 項目 | 内容 |
| --- | --- | --- |
| NFR-001 | 操作性 | Excelに近い時間帯別の見方を残しつつ、案件詳細へすぐ移動できる |
| NFR-002 | 保守性 | Java / Spring Bootの標準的な構成で、層を分けて実装する |
| NFR-003 | 拡張性 | 通知、地図連携、履歴管理などを後から追加しやすい構造にする |
| NFR-004 | セキュリティ | 実在の個人情報や顧客情報をサンプルデータに含めない |
| NFR-005 | 可用性 | ポートフォリオ段階では単一アプリ構成とし、停止時の復旧手順を明確にする |
| NFR-006 | 性能 | 一般的な月次スケジュール件数でストレスなく一覧表示できる |
| NFR-007 | テスト容易性 | ドメインロジックとサービス層を自動テストしやすくする |

## 将来拡張

MVP後に検討する機能は次の通り。

- 案件変更時のメール、チャット、アプリ通知
- Excelファイルからの案件取込
- ExcelまたはCSVへの出力
- Googleカレンダーなど外部カレンダー連携
- 地図表示、移動時間計算、ルート最適化
- 添付ファイル管理
- 作業完了報告、写真アップロード
- 承認フロー
- 詳細な監査ログ
- スマートフォン向けUIの強化

## 業務ルール初期案

- 案件には開始予定日時と終了予定日時を必須で持たせる
- 案件には少なくとも1名の主担当を設定する
- 車両は未定の状態も許可する
- 住所や顧客情報は案件詳細に集約する
- キャンセル済み案件は一覧に表示できるが、通常の予定とは区別する
- 作業完了後も案件データは履歴として残す

## 対象外

MVPでは次を対象外とする。

- 給与、勤怠、労務管理
- 実在顧客データの移行
- 外部サービスとの本格連携
- 高度な配車最適化
- 請求、売上、在庫管理

## 個人情報・機密情報の扱い

設計書、サンプルデータ、テストデータには、実在する会社名、人名、住所、電話番号、車両番号、顧客名を使わない。必要な場合は、次のような架空表現に置き換える。

- 顧客名: サンプル株式会社、テスト商事
- 担当者名: 社員A、担当者B
- 住所: 東京都サンプル区1-2-3
- 車両: 車両A、車両B
- 電話番号: 000-0000-0000

## 成功条件

- 担当者が当日の自分の案件と詳細情報をWeb上で確認できる
- 社員が案件登録時に必要情報を入力できる
- Excelセルに収まらない情報を案件詳細として管理できる
- MVPと将来拡張の境界が説明できる
- 要件、画面、DB、テスト方針が一貫している
